# 2.5D 等距游戏框架整体设计文档

## 一、 设计目标
- 提供一套可复用的、模块化的等距游戏基础架构。
- 支持菱形网格（60×30 菱形图块，经典等距投影，俯角 35.264°）。
- 包含坐标转换、深度排序、实体管理、输入处理、寻路与碰撞、资源加载等核心系统。
- 具备良好的调试支持，方便开发过程中快速定位问题。
- 易于扩展（添加新实体类型、新技能、新交互等）。

## 三、 核心模块详解

### 1. 坐标系统 (IsoUtils)
- 负责所有坐标转换：网格坐标 ↔ 世界坐标 ↔ 屏幕坐标。
- 提供基础公式，供其他模块调用。
- 输入：图块宽（60px）、高（30px）。
- 包含高度偏移计算：`screenY -= height * TILE_H / 2`（Z 轴厚度 10px 作为物体视觉高度，不影响坐标转换，但影响深度排序）。

### 2. 渲染管理 (RenderManager)
- 管理 Pixi.js 的舞台和图层。
- 动态深度排序：根据实体的屏幕 Y 坐标 + 高度偏移进行排序，保证遮挡正确。
- 支持视锥体剔除（仅渲染屏幕内物体）。
- 提供调试渲染：显示网格线、坐标文本、排序键值等。

### 3. 实体管理 (EntityManager)
- 所有游戏对象（角色、建筑、物品）继承自 `Entity` 基类。
- `Entity` 包含：网格坐标 (col, row)、高度 (height)、视觉精灵 (sprite)、尺寸、碰撞箱、是否可交互等。
- 实体更新：每帧根据位置自动更新屏幕坐标（由 `IsoUtils` 提供）。
- 支持动态添加/移除实体。

### 4. 地图与网格系统 (Map / Grid)
- 存储地形数据：每个格子的类型、高度、是否可行走、占用实体等。
- 提供寻路接口（A* 算法），基于 `walkable` 属性。
- 提供碰撞检测：根据格子是否可行走来判断移动合法性。
- 支持多层地图（如建筑内部、地下层）。

### 5. 输入系统 (Input)
- 统一处理鼠标、键盘、触摸事件。
- 鼠标点击：将屏幕坐标转换为网格坐标，并触发相应的交互（移动、拾取等）。
- 键盘控制：支持 WASD/方向键移动角色。
- 输入事件通过 `EventBus` 分发。

### 6. 资源管理 (ResourceManager)
- 负责加载纹理、音频、配置等。
- 使用 Pixi.js 的加载器（Loader）。
- 提供资源缓存和释放接口。

### 7. 场景管理 (SceneManager)
- 管理不同的游戏场景（如主菜单、游戏地图、设置界面）。
- 每个场景包含独立的渲染容器、实体集合、地图数据。
- 支持场景切换时的资源清理。

### 8. 事件总线 (EventBus)
- 全局事件通信，解耦模块。
- 例如：实体移动、点击选中、资源加载完成等。

## 四、 数据流与交互流程

### 典型流程：玩家点击地面移动
1. `InputSystem` 捕获鼠标点击坐标 (screenX, screenY)。
2. 调用 `IsoUtils.screenToGrid` 获取目标网格坐标 (col, row)。
3. 通过 `Map` 检查该格子是否可行走。
4. 若可行，调用 `PathFinder.findPath(start, end)` 获取路径点列表。
5. `Player` 接收路径，开始移动（逐帧插值）。
6. 每帧更新时，`RenderManager` 对所有实体重新排序并渲染。

## 五、 扩展性考虑

- **新实体类型**：只需继承 `Entity`，实现特定行为（如 `update()` 方法）。
- **新地形效果**：在 `Map` 中增加地形属性（如速度加成、陷阱）。
- **多玩家支持**：`EntityManager` 可容纳多个玩家实体，输入系统需要区分当前控制对象。
- **网络同步**：设计状态同步机制，所有实体位置变化通过 `EventBus` 广播。

## 组件之间的关系详解

2.5D 等距游戏框架中各组件通过明确的**依赖关系**和**事件驱动机制**协同工作。下面从三个层面分析它们的关系：**静态依赖**、**动态协作**和**关键接口**。

---

### 一、 静态依赖关系（分层结构）


**依赖方向**：上层依赖下层，核心层依赖工具层。具体表现为：
- `SceneManager` 持有 `RenderManager`、`EntityManager` 和 `Map` 的引用，负责创建和销毁场景。
- `RenderManager` 依赖 `IsoUtils` 进行坐标转换，依赖 `EntityManager` 获取所有需要渲染的实体，依赖 `Map` 获取地形数据（用于调试绘制）。
- `EntityManager` 依赖 `Map` 来验证位置合法性、获取地面高度等。
- `InputManager` 依赖 `IsoUtils` 将屏幕坐标转换为网格坐标，并通过 `EventBus` 触发交互事件。
- `PathFinder` 依赖 `Map` 的网格数据（walkable 数组）进行寻路。

---

### 二、 动态协作关系（运行时交互）

以**玩家点击移动**为例，完整的数据流如下：

1. **输入捕获**  
   `InputManager` 监听 Canvas 点击事件，获取屏幕坐标 `(sx, sy)`。

2. **坐标转换**  
   `InputManager` 调用 `IsoUtils.screenToGrid(sx, sy, originX, originY)`，得到目标网格坐标 `(targetCol, targetRow)`。

3. **事件发布**  
   `InputManager` 通过 `EventBus.emit('gridClick', { col, row })` 发布事件。

4. **逻辑处理**  
   `SceneManager` 或 `EntityManager` 订阅该事件。  
   - 获取当前玩家实体 `player` 的位置 `(playerCol, playerRow)`。  
   - 调用 `Map.isWalkable(targetCol, targetRow)` 检查目标是否可行走。  
   - 若可行，调用 `PathFinder.findPath(start, end)`，返回路径点数组 `path`。

5. **实体行为**  
   `player.moveAlongPath(path)` 被调用。  
   - `Player` 内部开始逐帧插值移动，每帧调用 `IsoUtils.gridToScreen` 更新 sprite 的屏幕位置。  
   - 每到达一个新格子，更新 `player.gridCol/gridRow`，并通知 `Map` 更新实体占用。

6. **渲染更新**  
   - `RenderManager` 在每帧的 `update` 中获取 `EntityManager` 的所有实体列表。  
   - 根据每个实体的 `screenY + height * TILE_H` 进行深度排序，得到渲染顺序。  
   - 调用 `LayerManager` 将实体按顺序添加到舞台容器中。  
   - `DebugUI` 若开启，会额外绘制网格线、坐标、排序键等。

7. **视效反馈**  
   - 移动过程中，`Player` 可能触发动画（通过 `ResourceManager` 加载的帧序列）。  
   - 若建筑遮挡玩家，`RenderManager` 在排序后检测并临时调整建筑透明度。

---

### 三、 关键接口与契约

组件之间的协作依赖于清晰的接口定义：

| 组件 | 对外提供的关键接口 | 依赖的接口 |
|------|------------------|-----------|
| **IsoUtils** | `gridToScreen()`, `screenToGrid()` | 无 |
| **Map** | `getTile(col, row)`, `isWalkable(col, row)`, `setEntity(col, row, entity)` | 无 |
| **EntityManager** | `addEntity(entity)`, `removeEntity(entity)`, `getAllEntities()`, `getPlayer()` | `Map` 用于验证位置 |
| **RenderManager** | `registerEntity(entity)`, `unregisterEntity(entity)`, `update(delta)` | `EntityManager`, `IsoUtils`, `Map`（调试用） |
| **PathFinder** | `findPath(start, end, walkableMatrix)` | `Map` 提供 `walkable` 数据 |
| **InputManager** | `init(canvas)`, `destroy()` | `IsoUtils`, `EventBus` |
| **EventBus** | `on(event, callback)`, `emit(event, data)`, `off(event, callback)` | 无 |
| **Entity** | `update(delta)`, `setPosition(col, row, height)`, `getScreenPosition()` | `IsoUtils` |

所有组件均通过 `EventBus` 进行松耦合通信，避免直接依赖。例如：
- 当 `Map` 数据变化时（如地形改变），触发 `mapChanged` 事件，`RenderManager` 监听后重绘地形。
- 当玩家生命值变化时，`Player` 触发 `playerHpChanged` 事件，UI 系统响应更新血条。

---

### 四、 依赖注入与生命周期管理

为了便于测试和扩展，框架采用**依赖注入**模式（通过构造函数传入依赖）：
```typescript
class RenderManager {
    constructor(
        private app: PIXI.Application,
        private entityManager: EntityManager,
        private map: Map,
        private isoUtils: typeof IsoUtils
    ) {}
}
```

生命周期管理由 `SceneManager` 统一负责：
- 切换场景时，`SceneManager` 销毁旧的 `RenderManager`、`EntityManager`、`Map` 等实例，创建新实例。
- 资源加载通过 `ResourceManager` 异步进行，完成后触发 `resourcesLoaded` 事件，通知各模块开始渲染。

---

### 五、 总结

组件之间的关系可以概括为：
- **分层清晰**：工具层提供基础能力，核心层组合工具实现功能，应用层管理整体流程。
- **松耦合**：通过事件总线解耦输入、逻辑、渲染，新增功能无需修改现有组件。
- **单向依赖**：高层依赖低层，避免循环依赖。
- **接口契约**：每个组件只依赖明确的外部接口，便于替换实现（如更换寻路算法）。

这种架构确保了等距游戏框架的**可维护性**、**可扩展性**和**可测试性**，为后续游戏逻辑开发提供了稳定基础。

## UI 组件补充设计

### 二、 UI 模块职责

#### 1. UIManager
- **生命周期管理**：创建、显示、隐藏、销毁 UI 组件。
- **层级控制**：维护多个渲染层（如底层 HUD、中层弹窗、顶层提示），确保正确的遮挡和输入优先级。
- **输入穿透**：当 UI 打开时，决定点击事件是否穿透到等距世界（例如点击背包时不应移动角色）。
- **资源管理**：负责加载 UI 纹理、字体等资源（可复用 ResourceManager）。

#### 2. UIComponent (基类)
所有 UI 控件的抽象基类，定义通用属性和方法：
```typescript
abstract class UIComponent extends PIXI.Container {
    protected manager: UIManager;
    protected visible: boolean = true;
    protected interactive: boolean = true;

    constructor(manager: UIManager) {
        super();
        this.manager = manager;
    }

    abstract show(): void;
    abstract hide(): void;
    abstract update(delta: number): void; // 可选，用于动画
}
```

#### 3. 具体控件
- **Button**：支持点击、悬停状态，发出 `buttonClick` 事件。
- **Panel**：带背景、边框的容器，可拖动。
- **ProgressBar**：用于血条、经验条。
- **Text**：动态文本标签，支持本地化。
- **Minimap**：小地图，显示等距世界的地形和单位。
- **Tooltip**：悬停提示。
- **Dialog**：模态对话框，带确认/取消按钮。
- **HUD**：常驻信息条（角色头像、生命值、金币、背包按钮等）。

### 三、 UI 与核心系统的交互

UI 通过 **EventBus** 与游戏核心模块通信，避免直接耦合：

| 事件名 | 触发者 | 监听者 | 说明 |
|--------|--------|--------|------|
| `playerHpChanged` | Player | HUD | 更新血条显示 |
| `playerGoldChanged` | Inventory | HUD | 更新金币数字 |
| `uiButtonClick` | Button | UIManager → 转发给游戏逻辑 | 例如打开背包、移动角色等 |
| `showDialog` | 游戏逻辑 | Dialog | 弹出确认框 |
| `minimapClick` | Minimap | 寻路系统 | 点击小地图移动到对应位置 |

#### 示例：血条更新
```typescript
// 在 Player 中
this.eventBus.emit('playerHpChanged', { currentHp, maxHp });

// 在 HUD 中
this.eventBus.on('playerHpChanged', (data) => {
    this.hpBar.setProgress(data.currentHp / data.maxHp);
});
```

### 四、 输入处理与 UI 穿透

等距游戏的点击操作需要区分“点击 UI”和“点击游戏世界”。解决方案：
1. **UIManager 维护一个“阻塞输入”标志**：当有模态对话框打开时，所有点击事件不传递给 InputManager。
2. **使用事件冒泡机制**：Canvas 上的点击事件先由 UIManager 处理，若没有 UI 消费，再转发给 InputManager。
3. **通过 Pixi 的 `interactive` 和 `hitArea`**：UI 组件可设置 `interactive = true`，Pixi 会自动判断鼠标是否落在 UI 上，从而阻止事件传播。

```typescript
// 在 UIManager 中
onCanvasClick(e: PIXI.InteractionEvent) {
    // 如果 UI 正在处理模态弹窗，直接拦截
    if (this.modalActive) {
        e.stopPropagation();
        return;
    }
    // 否则交给 InputManager 处理世界点击
    this.inputManager.handleWorldClick(e);
}
```

### 五、 UI 资源配置

UI 资源（图片、字体、样式表）统一由 `ResourceManager` 加载，`UIManager` 在初始化时获取。建议使用 **TexturePacker** 将 UI 素材打包成图集，提升性能。

```typescript
class UIManager {
    private resources: ResourceManager;
    private textures: Map<string, PIXI.Texture>;

    constructor(resources: ResourceManager) {
        this.resources = resources;
        this.textures = resources.getTextures('ui_atlas'); // 获取图集
    }
}
```

### 六、 典型 UI 控件设计示例

#### 1. 按钮 (Button)
```typescript
class Button extends UIComponent {
    private background: PIXI.Sprite;
    private label: PIXI.Text;
    private onClick: () => void;

    constructor(manager: UIManager, x: number, y: number, text: string, onClick: () => void) {
        super(manager);
        this.onClick = onClick;
        this.background = new PIXI.Sprite(manager.textures['button_bg']);
        this.label = new PIXI.Text(text, { fill: 0xffffff });
        this.addChild(this.background, this.label);
        this.position.set(x, y);
        this.interactive = true;
        this.on('pointertap', this.handleClick, this);
    }

    private handleClick() {
        this.onClick();
        this.manager.eventBus.emit('buttonClick', { button: this });
    }
}
```

#### 2. 血条 (ProgressBar)
```typescript
class ProgressBar extends UIComponent {
    private fillBar: PIXI.Sprite;
    private maxWidth: number;

    constructor(manager: UIManager, x: number, y: number, width: number, height: number) {
        super(manager);
        this.maxWidth = width;
        const bg = new PIXI.Graphics().beginFill(0x333333).drawRect(0, 0, width, height);
        this.fillBar = new PIXI.Sprite(PIXI.Texture.WHITE);
        this.fillBar.tint = 0xff0000;
        this.fillBar.width = width;
        this.fillBar.height = height;
        this.addChild(bg, this.fillBar);
        this.position.set(x, y);
    }

    setProgress(percent: number) {
        this.fillBar.width = Math.max(0, this.maxWidth * Math.min(1, percent));
    }
}
```

### 七、 UI 动画与过渡

为增强体验，UI 应支持简单动画（淡入淡出、滑动）。可利用 Pixi 的 `Tween` 库或自行实现基于 `requestAnimationFrame` 的插值。

```typescript
class UIComponent {
    async fadeIn(duration: number = 200): Promise<void> {
        this.alpha = 0;
        this.visible = true;
        return new Promise(resolve => {
            const start = performance.now();
            const animate = (now: number) => {
                const elapsed = now - start;
                this.alpha = Math.min(1, elapsed / duration);
                if (elapsed < duration) requestAnimationFrame(animate);
                else resolve();
            };
            requestAnimationFrame(animate);
        });
    }
}
```

### 十、 总结

UI 组件作为独立模块，通过事件总线与游戏核心交互，保持了整体架构的清晰和可扩展性。UIManager 负责统一管理和输入穿透，基础控件提供了可复用的 UI 构建块，动画和调试功能进一步提升了开发体验。这套设计可以支撑从简单 HUD 到复杂 RPG 界面的各种需求。
