# Chrome 模拟方案

核心策略是：**保留硬件层的真实指纹，仅移除软件层的个性化特征，并通过确定性噪声打破 Canvas 追踪，同时严格保持所有信号的自洽**。

- 使用高斯噪声：基于 Box-Muller 变换，将均匀随机数转换为高斯分布。指定标准差 σ ≈ 0.8，这样大部分噪声集中在 -1 到 +1 之间，极少数情况出现 ±2。

- 上下文感知的噪声密度：分析 Canvas 命令流。如果检测到 createLinearGradient 或 shadowBlur，增加该区域的噪声概率和幅度；如果是纯色 fillRect，降低噪声概率甚至完全跳过。这需要解析 Canvas 绘制命令队列。

- 噪声频率与 GPU 型号绑定：不同架构的 GPU，其“高频抖动”和“低频色带”的表现不同。你可以预先针对你的 Intel UHD 620 采集 500 次 Canvas 渲染样本，训练一个小型生成模型（GAN 或 VAE），然后根据 profileID 生成符合该 GPU 纹理的噪声模式。

---

## 1. 信号链自洽模型

你需要让浏览器暴露的所有属性都能对得上同一台“标准宿主机”：

| 指纹维度 | 策略 | 实现方式 |
|----------|------|----------|
| 操作系统/版本 | 保持真实 | User-Agent、`navigator.platform`、`navigator.userAgentData` 中的平台字段完全使用宿主机的值，不修改。 |
| GPU 型号/驱动 | 保持真实 | WebGL 的 `UNMASKED_VENDOR_WEBGL`、`UNMASKED_RENDERER_WEBGL` 以及 Canvas 渲染特征均基于真实 GPU，不做任何伪装。 |
| CPU/内存 | 保持真实 | `hardwareConcurrency`、`deviceMemory` 等不做修改（若宿主机属于常见配置，则天然融入群体）。 |
| 屏幕 | 轻微修正 | 分辨率可使用真实值减去任务栏高度（Windows 40px），以匹配普通窗口化浏览行为；`devicePixelRatio` 保持真实。 |
| 字体列表 | **减少（裁剪）** | 仅暴露操作系统自带的默认字体 + 极少量高普及率字体（如 Office 核心字体），隐藏所有专业软件、开发工具、游戏带来的独家字体。 |
| 浏览器插件 | 清空/固定 | 将 `navigator.plugins` 和 `navigator.mimeTypes` 重置为标准 Chrome 内置插件（PDF Viewer、Chrome PDF Plugin 等），移除任何第三方插件。 |
| 时区/语言 | 保持真实（匹配 IP） | 使用宿主机时区和首选语言，但需确保与实际出口 IP 地理定位一致。 |
| Canvas 指纹 | **确定性地改变** | 基于 `profileID` 注入极微小的亚像素噪声（仅扰动最低有效位），使得不同 profile 产生不同的 Canvas 哈希，但噪声模式必须符合真实 GPU 的误差分布，避免产生“伪造痕迹”。 |
| AudioContext | 同样注入确定性噪声 | 对 DynamicsCompressor 的输出乘以 `1 + ε(profileID)`（ε ≈ 1e-7），保持与 Canvas 一致的伪装身份。 |

只要所有信号均指向同一套“真实硬件 + 精简软件”的组合，反检测系统就会将其归为正常设备，而不会触发异常警报。

## 2. 字体裁剪：只减不增，且不破坏系统字体

字体是最容易泄露用户身份的信号之一，你的要求是保留系统默认字体，移除所有第三方软件带来的额外字体。

### 2.1 实现原理
- 拦截所有字体枚举 API：`document.fonts`（`FontFaceSet`）、`navigator.fonts.query()`（需用户授权）、CSS 字体探测（`document.fonts.check()`）。
- 在页面脚本执行前注入过滤逻辑，使暴露的字体列表仅包含 **Win10/Win11 默认字体** + 可选的高覆盖字体（如微软雅黑、宋体、Arial、Times New Roman 等），并完全隐藏其他字体。
- 不实际删除系统字体文件，因此浏览器底层仍能用被隐藏的字体进行渲染，但字体探测脚本会发现“该字体不存在”。这样，Canvas 绘制文字时若指定了被隐藏的字体，浏览器会使用回退字体，从而微妙地改变 Canvas 指纹，这也是一种合法的特征变动。

### 2.2 代码示例（Puppeteer/CDP 脚本注入）
```javascript
// 在页面创建文档前注入
await page.evaluateOnNewDocument(() => {
  // 允许的字体白名单
  const WHITELIST = new Set([
    'arial', 'times new roman', 'courier new', 'verdana', 'tahoma',
    'microsoft yahei', 'simsun', 'simhei', 'mingliu', 'gulim', 'malgun gothic'
  ]);

  // 拦截 FontFaceSet.prototype.has/check
  const origCheck = FontFaceSet.prototype.check;
  FontFaceSet.prototype.check = function(font, text) {
    const family = font.split(' ')[0].replace(/['"]/g, '').toLowerCase();
    if (!WHITELIST.has(family)) return false;
    return origCheck.call(this, font, text);
  };

  // 拦截 queryLocalFonts (如果启用)
  if (navigator.fonts && navigator.fonts.query) {
    const origQuery = navigator.fonts.query;
    navigator.fonts.query = async function() {
      const allFonts = await origQuery.call(this);
      return allFonts.filter(f => WHITELIST.has(f.family.toLowerCase()));
    };
  }
});
```

**注意**：此方法无法阻止通过 `<canvas>` 测量文字宽度的间接探测，但因为你已裁剪了字体列表，探测脚本会发现被隐藏字体“不可用”，从而不会使用它们进行指纹计算，这本身就改变了指纹且不产生矛盾。

## 3. Canvas 确定性扰动：符合真实 GPU 特征的亚像素噪声

你保留了真实 GPU，因此 Canvas 渲染结果与所有同型号 GPU 用户相同。为了打破唯一性，你需要注入噪声，但又不能引入“非该 GPU 特征”的错误模式。

### 3.1 噪声设计原则
- **确定性**：噪声 = f(profileID, 像素坐标)，同一 profile 下每次绘制结果完全一致。
- **微小性**：仅修改像素值的最低 1~2 位，幅度控制在 ±2 以内，人眼不可见。
- **符合 GPU 误差分布**：不同 GPU 型号在浮点精度、抗锯齿等方面有特定的误差特征。例如，你的真实 GPU 是 Intel UHD Graphics 620，那么它的 Canvas 噪声通常集中在边缘像素的 alpha 通道上，且数值变化遵循类似高斯分布的模式。你需要**采集该 GPU 的基线噪声**，然后让注入的噪声遵循相同的分布。

### 3.2 基线采集与建模
你可以使用之前我提供的采集工具，在同一台机器上多次运行同一 Canvas 测试（如绘制贝塞尔曲线），每次重启浏览器或使用不同的“干净 profile”，观察像素值的波动（真实波动很小，通常在 ±1 内）。记录下这些波动发生的像素位置和幅度分布，构建一个特征模型。

### 3.3 注入实现
利用 Canvas 劫持，在 `toDataURL` / `getImageData` 被调用时，按预存的特征分布修改像素：
```javascript
// 伪代码
function applyProfileNoise(imageData, profileSeed) {
  const rng = seedrandom(profileSeed);
  for (let i = 0; i < data.length; i += 4) {
    // 以极低概率修改该像素，模拟真实 GPU 的随机波动
    if (rng() < 0.002) { // 概率可调
      data[i]   = clamp(data[i]   + rngInt(rng, -1, 1));
      data[i+1] = clamp(data[i+1] + rngInt(rng, -1, 1));
      data[i+2] = clamp(data[i+2] + rngInt(rng, -1, 1));
    }
  }
}
```
**关键**：这个噪声必须与你的 GPU 型号在其他信号上保持一致。不能出现“Canvas 呈现 Intel 核显的低频噪声”而 WebGL 却报告为“NVIDIA 独立显卡”的矛盾。

## 4. 避免信号矛盾的集成检查清单

模拟完成后，必须逐项验证以下信号全部匹配，否则就是严重破绽：

- [ ] User-Agent 中的平台字段（Windows NT 10.0）与 `navigator.platform`（Win32）匹配。
- [ ] WebGL `UNMASKED_RENDERER_WEBGL` 字符串与实际 GPU 一致（例如 `ANGLE (Intel, Intel(R) UHD Graphics 620 Direct3D11 vs_5_0 ps_5_0)`）。
- [ ] Canvas 渲染哈希在同 profile 下稳定，且其噪声模式符合 Intel UHD 的特征（如果你用的是这台 GPU）。
- [ ] AudioContext 指纹与 Canvas 噪声使用相同的 profileID 种子，且波动幅度一致。
- [ ] 屏幕分辨率、色深、像素比与实际窗口化设置匹配，不能是 3840×2160 却搭配 96 DPI。
- [ ] 字体列表仅包含白名单内的字体，且其中不含有任何与当前 GPU/系统不协调的字体（例如 Mac 专用字体出现在 Windows 上）。
- [ ] 时区、语言、键盘布局与出口 IP 地理定位不冲突（可以用 `Intl.DateTimeFormat().resolvedOptions()` 验证）。

## 5. 高级实现：Chromium 源码级修改（可选）

如果想要完全避免 JS 注入可能带来的原型链篡改痕迹，你可以直接修改 Chromium 源码，在底层实现：

- **字体枚举过滤**：在 `third_party/blink/renderer/platform/fonts/` 中修改 `FontCache` 或 `FontEnumeration` 相关逻辑，使系统返回的字体列表只包含白名单。
- **Canvas 噪声注入**：在 `third_party/blink/renderer/platform/graphics/canvas/` 的 `ImageDataBuffer` 或 `CanvasResourceProvider` 处，根据 `profileID` 叠加噪声，这样连 JS 都无法感知到修改。

这种方法隐蔽性最高，但需要维护自己的 Chromium 分支，适合专业反检测浏览器开发。

## 6. 总结：最终效果

通过上述设置，Chrome 将呈现出一个“和宿主机一模一样的硬件，但软件环境极其干净、且 Canvas 指纹带着一丝独特的、但符合该硬件自然波动的噪声”的形象。这样的信号链既不会因伪装成另一款 GPU 而被反指纹系统标记为异常，又能通过字体裁剪和确定性噪声打破跨站点追踪，实现“一人千面但面面相似”的隐匿效果。

# 一套**纯程序化、无模型依赖**的 Canvas 指纹噪声注入方案

使用 PRNG + 低频柏林噪声模拟芯片级全局误差，并对边缘区域施加符合目标 GPU（以 Intel 核显为例）偏差特征的卷积核。所有操作均基于 `profileID` 保证确定性，适合作为初步对抗方案的快速实现。

---

## 1. 总体流程

```
profileID → SHA-256 → 128-bit seed → PRNG (ChaCha20)
                │
                ▼
        低频柏林噪声生成 (3层叠加)
                │
                ▼
        全局噪声图 (W×H, 幅度 ±2)
                │
                ▼
        原像素 + 全局噪声 + 边缘卷积核补偿 → 输出
```

- **全局噪声**：模拟 GPU 制造工艺、电压波动等引起的低频误差（如色带、块状微亮暗）。
- **边缘卷积核**：模拟 Intel 核显在 Direct3D 驱动下边缘抗锯齿偏暗、像素点偏大等特征。
- **确定性**：同一 `(profileID, canvas宽, canvas高, 操作类型哈希)` 生成完全相同的噪声图。

---

## 2. 低频柏林噪声生成 (C++ 伪代码)

使用 3 层柏林噪声（或改进的 Simplex 噪声）叠加，形成具有空间连续性的低频波动。每层缩放因子（`scale`）和振幅（`amplitude`）不同：

| 层 | 缩放因子 | 振幅（像素值偏移量） |
|----|----------|----------------------|
| 1  | 0.05     | ±2                   |
| 2  | 0.1      | ±1                   |
| 3  | 0.2      | ±0.5                 |

总偏移为三层叠加的加权和。这种设计使得噪声在几个像素到几十像素范围内缓慢起伏，类似芯片版图不均匀导致的微色调差异。

```cpp
// 确定性 PRNG (ChaCha20, seed from profileID)
class ProfiledPRNG {
  uint64_t state_;
  // 基于 seed 初始化 ...
public:
  uint64_t Next() { /* ... */ }
  double NextDouble() { return Next() / static_cast<double>(UINT64_MAX); }
};

// 2D Simplex 噪声实现 (略，标准算法)
double SimplexNoise(double x, double y, ProfiledPRNG& prng);

void GenerateLowFreqNoise(uint8_t* pixel_data, int width, int height,
                          uint64_t seed) {
  ProfiledPRNG prng(seed);
  for (int y = 0; y < height; ++y) {
    for (int x = 0; x < width; ++x) {
      double nx = x * 0.05, ny = y * 0.05;
      double v = 0.0;
      // 三层叠加，每层用 PRNG 改变噪声函数的哈希表偏移
      v += SimplexNoise(nx, ny, prng) * 2.0;          // 层1
      v += SimplexNoise(nx * 2, ny * 2, prng) * 1.0;  // 层2
      v += SimplexNoise(nx * 4, ny * 4, prng) * 0.5;  // 层3
      // 映射到整数偏移 [-2, 2]
      int offset = static_cast<int>(std::round(v));
      offset = std::clamp(offset, -2, 2);
      // 存储偏移量到噪声图
      noise_r[y * width + x] = offset;
      // G/B 通道可以相同或独立 PRNG 分支
    }
  }
}
```

**要点**：`SimplexNoise` 的实现中，排列表（permutation table）由 `ProfiledPRNG` 生成，从而绑定 seed，使得不同 profile 产生完全不同的低频波动图案。

---

## 3. 边缘偏差模拟：Intel 核显暗边卷积核

Intel UHD Graphics 6xx 系列在 Direct3D 驱动下，路径边缘的像素渲染值通常比理论值**偏暗 2~3 个色阶**，且边缘梯度略宽（亚像素覆盖范围更大）。我们可以通过一个 3×3 或 5×5 卷积核在检测到的边缘区域施加补偿：

### 3.1 边缘检测
使用 Sobel 算子计算亮度梯度幅值，并设定阈值区分平滑区和边缘区。

### 3.2 卷积核定义
对于被判定为边缘的像素，应用以下 3×3 核：

```
R 通道 (偏暗):   -2/16   -2/16   -2/16
                -2/16   +8/16   -2/16
                -2/16   -2/16   -2/16

G 通道:           -1/16   -1/16   -1/16
                -1/16   +4/16   -1/16
                -1/16   -1/16   -1/16

B 通道:           -1/16   -1/16   -1/16
                -1/16   +4/16   -1/16
                -1/16   -1/16   -1/16
```

**效果**：边缘像素整体亮度轻微下降（R 通道降幅稍大），且产生邻域扩散，模拟了 Intel 驱动下字体和图形边缘的“肉感”渲染特征。

### 3.3 应用方式
仅对梯度幅值大于阈值（如 20）的像素应用该卷积核。为避免过度平滑，边缘像素重复应用 1 次即可。

```cpp
void ApplyEdgeDarkening(uint8_t* image, int width, int height) {
  // 灰度图生成
  std::vector<uint8_t> gray(width * height);
  for (int i = 0; i < width * height; ++i) {
    gray[i] = 0.299 * image[i*4] + 0.587 * image[i*4+1] + 0.114 * image[i*4+2];
  }
  // Sobel 梯度计算
  // ... 使用标准 Sobel 得到 grad_mag
  std::vector<float> grad(width * height);
  // ...

  std::vector<uint8_t> orig = image; // 拷贝原图
  for (int y = 1; y < height-1; ++y) {
    for (int x = 1; x < width-1; ++x) {
      if (grad[y*width + x] < 20) continue; // 非边缘跳过
      for (int c = 0; c < 3; ++c) {
        float sum = 0;
        for (int dy = -1; dy <= 1; ++dy) {
          for (int dx = -1; dx <= 1; ++dx) {
            int idx = ((y+dy)*width + (x+dx)) * 4 + c;
            float weight = kernel[c][dy+1][dx+1]; // kernel 3x3 for each channel
            sum += orig[idx] * weight;
          }
        }
        image[(y*width + x)*4 + c] = Clamp(static_cast<int>(sum));
      }
    }
  }
}
```

**注意**：卷积核参数应由 `profileID` 决定（例如在范围内随机选取偏暗程度和扩散半径），避免所有伪装环境边缘特征完全相同。

---

## 4. 融合与性能优化

- **噪声缓存**：以 `(seed, width, height)` 为 key 缓存全局噪声图，相同 profile 相同尺寸 Canvas 直接复用。
- **边缘检测缓存**：可以缓存原始 Canvas 的梯度图吗？不行，因为绘制内容可能不同。但若内容哈希相同则可复用。
- **分块处理**：大画布（>4000px）可分块处理，减少内存峰值。

---

## 5. 局限性 & 与真实 GPU 统计特征的差距

- **低频噪声**虽然模拟了全局渐变，但缺少高频小尺度抖动（像素级独立性）。真实 GPU 的误差在像素间更独立（白噪声成分），可加入一层确定性白噪声（PRNG 直接生成每个像素的 ±1）模拟。
- **边缘卷积**难以完美匹配 Intel 驱动精确的像素着色器行为，尤其在不同字体下，字符轮廓的边缘效果可能不一致。
- **不会自动适应不同 GPU**：目标特征是硬编码的 Intel 核显模式，若实际宿主机是 NVIDIA，这种“Intel 风格”可能导致异常。

**应对**：可以同时采集主流 GPU（Intel UHD / NVIDIA GeForce / AMD Radeon）的偏差参数，根据配置选择对应的卷积核和低频幅度。

---

## 6. Chromium 集成要点

与之前 GAN 方案一样，在 `CanvasResourceProvider::ProduceCanvasResource()` 或 `StaticBitmapImage` 构造处注入，直接修改像素缓冲。使用 `base::SHA256` 将 profileID 转为 seed，调用 PRNG 函数生成噪声图。

**性能开销**：全图低频噪声生成 + 边缘滤波耗时约为 `O(W*H)`，对于 1920×1080 的 Canvas 大约 2-3 ms（单核 C++ 优化代码），可接受。

---

## 7. 总结

本程序化方案用 **低频柏林噪声 + 边缘卷积核** 模拟了 Intel 核显的典型指纹特征，相比神经网络模型更简单、可解释，且不需要额外资源文件。核心优势是：
- 完全确定性，绑定 profileID。
- 内存和 CPU 开销极低，可内置在 Chromium 中。
- 可通过参数调整（噪声振幅、卷积核权重）快速适应不同 GPU 配置。

但它是**近似**的，缺乏真实硬件的高阶统计细节，适合作为反指纹系统的中等强度对抗方案，或在资源受限环境下快速上线。对于高强度商用场景，仍需升级为基于真实数据训练的生成模型。

# 嵌入轻量生成模型

其核心思想是：用目标 GPU 的真实渲染误差分布，替换掉宿主 GPU 的固有误差，使得最终输出的像素数据在统计上看起来来自另一台同型号的设备，且同一 profile 下每次渲染结果完全一致。

---

## 1. 方案总体架构

```
┌─────────────── ProfileID ───────────────┐
│                                         │
▼                                         ▼
种子 PRNG (确定性)                 GPU 特征模型 (VAE/GAN)
│                                         │
▼                                         ▼
生成空间噪声图 (弥漫性高斯)        调整噪声风格 (目标硬件倾向)
│                                         │
└──────────────────┬──────────────────────┘
                   ▼
        原 Canvas 像素 → 融合 → 输出像素 (toDataURL/getImageData)
```

- **确定性**：`profileID` 经过哈希后作为种子驱动伪随机数生成器，并作为生成模型的条件输入，保证每次相同输入产生相同噪声。
- **隐蔽性**：噪声模式由生成模型产生，该模型已经学会了目标 GPU 的误差分布（如 Intel UHD 620 的特定抖动特征），因此像素波动完全符合该硬件的统计指纹，不会被归类为“伪造”。

---

## 2. 目标 GPU 误差数据采集与模型训练

### 2.1 采集原始数据
在真实的 **目标型号 GPU + 特定 OS** 的机器上，运行一套标准的 Canvas 测试用例（类似之前提供的特征采集器），这些用例覆盖了：
- 纯色填充 + 渐变
- 各种字体和字号的文字渲染
- 复杂路径（贝塞尔曲线、弧线）
- 阴影、模糊滤镜
- 透明度和全局合成操作

对每个测试用例，同时记录：
- **输入**：绘制命令序列（或离屏 Canvas 的像素缓冲，在“无噪声”参考下）。
- **输出**：真实 GPU 渲染后的像素数据。

为了获得“误差分布”，需要一个**无噪声基准**：可以使用 CPU 软件渲染（如 Skia 的软件后端）生成同一场景的像素值，或者使用多台同型号 GPU 的平均值作为基准。然后计算 `误差图 = 真实像素 - 基准像素`。

### 2.2 训练轻量生成模型
推荐使用**条件变分自编码器（CVAE）** 或**条件 GAN（cGAN）**，因为：
- VAE 学习潜在分布，可以平滑地根据条件（种子）生成符合分布的误差。
- 模型输入可以包含：原像素的颜色值、像素坐标、周围像素信息（局部感受野），以及一个噪声种子（从 profileID 派生的随机向量）。
- 输出：每个像素的 RGB 偏移量。

为了嵌入 Chromium，模型必须极小（<200KB）。可采用：
- 极度简化的网络结构（全卷积，层数少，通道数小）。
- 使用 8-bit 量化（TensorFlow Lite 的 int8 模型）。
- 限制输入尺寸，如一次只处理 16×16 的 tile，然后全图滑动（或使用全卷积，一次处理整张图但内存开销小）。

训练完成后，导出为 **TensorFlow Lite FlatBuffer** 文件。该文件将被编译进 Chromium 的资源段（或 base64 后嵌入 C++ 数组）。

## 3. Chromium 源码修改位置与实现

### 3.1 关键修改点
Canvas 像素最终汇聚到两个出口：
- `CanvasRenderingContext2D::getImageData()` → 调用 `GetImageData` 内部。
- `HTMLCanvasElement::toDataURL()` → 最终通过 `CanvasResourceProvider` 获取快照。

最合适的注入点是在**位图数据即将返回给调用者之前**，位于：
```
src/third_party/blink/renderer/platform/graphics/canvas/
  canvas_resource_provider.cc
  (或 static_bitmap_image.cc)
```
在创建 `StaticBitmapImage` 或从 `CanvasResourceProvider::ProduceCanvasResource()` 返回的 `scoped_refptr<StaticBitmapImage>` 中，其内部持有像素缓冲（`PaintImage`）。我们可以在构造该图像后，立即在内存中修改其像素。

另一个更底层的点是 `ImageDataBuffer`，但为避免影响其他模块，建议在 Canvas 专属路径中注入。

### 3.2 确定性噪声注入流程
1. 每个浏览器 Context（对应一个 profile）在初始化时，根据 `profileID` 计算出一个 128-bit 的 `noise_seed`（可通过 SHA-256 截取）。
2. 在 Canvas 快照生成路径中，获取当前 profile 的 seed。
3. 使用该 seed 初始化一个局部 PRNG，并结合 canvas 尺寸、操作类型等，生成一个控制向量。
4. 加载嵌入的 TensorFlow Lite 模型，运行推理，生成全图的像素偏移图（或 tile-by-tile 推理）。
5. 将偏移图叠加到原始像素上，并确保结果裁剪在 [0,255] 范围内。

**缓存优化**：对于同一个 profile，相同尺寸和相同“绘制特征”（如内容哈希）的 canvas，噪声模式应当相同，因此可以缓存 `(seed, width, height, content_hash) → noise_map` 的映射，避免每帧推理。

### 3.3 代码修改示例（伪代码）
```cpp
// 在 canvas_resource_provider.cc 或类似位置
scoped_refptr<StaticBitmapImage>
CanvasResourceProvider::ProduceCanvasResource() {
  // ... 原有逻辑获得 image
  scoped_refptr<StaticBitmapImage> image = ...;
  
  // 注入噪声 (前提：当前 profile 启用了伪装)
  if (ShouldApplyFingerprintNoise()) {
    ApplyNoiseToImage(image, GetProfileNoiseSeed());
  }
  return image;
}

void ApplyNoiseToImage(scoped_refptr<StaticBitmapImage> image,
                      uint64_t seed) {
  // 获取像素可写缓冲区
  SkPixmap pixmap;
  if (!image->PeekPixmap(&pixmap)) return;
  
  int width = pixmap.width();
  int height = pixmap.height();
  
  // 检查缓存
  std::string cache_key = ComputeCacheKey(seed, width, height,
                                           image->GetContentHash());
  std::vector<float> noise_map;
  if (!TryGetCachedNoise(cache_key, &noise_map)) {
    // 运行模型推理
    noise_map = GenerateNoiseMap(seed, width, height);
    CacheNoise(cache_key, noise_map);
  }
  
  // 叠加噪声
  for (int y = 0; y < height; ++y) {
    for (int x = 0; x < width; ++x) {
      int idx = (y * width + x) * 4;
      uint8_t* pixel = pixmap.writable_addr8(x, y);
      pixel[0] = Clamp(pixel[0] + noise_map[idx + 0]);
      pixel[1] = Clamp(pixel[1] + noise_map[idx + 1]);
      pixel[2] = Clamp(pixel[2] + noise_map[idx + 2]);
      // Alpha 不变
    }
  }
}
```

### 3.4 模型推理模块集成
需要将 TensorFlow Lite 的 C++ 库链接到 Chromium。Chromium 本身已经包含 TFLite（用于一些 AI 功能），可以直接使用。或者使用更轻量的 `USearch` 或纯 C 实现的推理器，但 TFLite 是最便捷的选择。

在 `third_party/blink/renderer/platform/graphics/canvas/` 下新建 `fp_noise_model.h/cc`，负责：
- 一次性从资源加载模型。
- 提供推理接口：接受 seed 和尺寸，返回噪声 map。
- 利用 `tflite::FlatBufferModel` 和 `tflite::Interpreter` 执行推理。

模型输入格式：一个包含随机噪声向量 (1×64) 和可选坐标网格的 tensor，输出为 (1×H×W×3) 的偏移量 tensor（或仅输出低频分量，然后上采样）。

由于 Canvas 尺寸可能很大，可采用**拉普拉斯金字塔分解**：模型只生成 64×64 的低频噪声，然后上采样并与高频随机噪声（由 PRNG 根据 seed 生成，但风格固定）混合。这样既保证了生成噪声的全局风格（低频结构），又通过 PRNG 保证了高频细节的确定性。

## 4. 配套修改：跨信号一致性

仅 Canvas 噪声还不够，必须同步修改：

- **WebGL 渲染器字符串**：在 `third_party/blink/renderer/modules/webgl/webgl_rendering_context_base.cc` 中，拦截 `getParameter(UNMASKED_VENDOR_WEBGL)` 和 `UNMASKED_RENDERER_WEBGL`，返回目标 GPU 的相应字符串（如 `"Google Inc. (Intel)"` 和 `"ANGLE (Intel, Intel(R) UHD Graphics 620)"`）。
- **AudioContext 指纹**：在 `third_party/blink/renderer/modules/webaudio/` 的 `DynamicsCompressorNode` 或 `AudioBuffer` 处理处，根据同一 seed 对样本进行确定性微调。
- **字体列表**：在 `third_party/blink/renderer/platform/fonts/` 修改字体枚举逻辑，仅返回预定义的基础字体集。
- **屏幕、CPU 内存等**：在 `third_party/blink/renderer/core/frame/navigator*` 相关文件中设置固定值或基于 seed 从预定义集合中选择。

所有这些修改都使用同一个 `profileID` 派生的 seed，确保逻辑自洽。

## 5. 性能与隐蔽性分析

### 5.1 性能开销
- 模型推理：轻量模型推理一次（例如 64×64→上采样）耗时 < 1ms，加上缓存后几乎零开销。
- 像素混合：直接内存操作，O(W*H)，但只在 Canvas 快照时触发，频率极低。

### 5.2 隐蔽性
- **JS 层无感知**：所有修改在 C++ 层完成，原型链、函数 toString 均保持原生，无法从 JavaScript 检测。
- **统计自然**：生成的噪声通过真实数据训练，分布与目标 GPU 一致，ML 分类器会将其视为正常硬件差异。
- **确定性稳定**：同一 profile 的噪声永不变化，通过反复渲染测试不会暴露随机性。

## 6. 挑战与应对

| 挑战 | 应对措施 |
|------|----------|
| 模型训练需要大量目标 GPU 数据 | 可先针对最常用的 GPU（如 Intel UHD，市场占有率 >30%）制作模型，覆盖大多数场景。 |
| 不同浏览器版本可能改变渲染细节 | 模型需随 Chromium 版本更新而重新校准，可将模型作为外部数据包，方便更新。 |
| 模型文件体积可能增大二进制 | 使用极度压缩的模型（<100KB），或只保存统计参数，运行时用程序化噪声合成。 |
| 推理延迟可能影响交互 | 使用缓存 + 异步生成，或预生成噪声图，对于需要实时绘制的动画 Canvas，可暂时禁用噪声。 |

---

## 7. 替代方案：程序化噪声 + 风格迁移

如果不便嵌入神经网络，也可使用**程序化噪声生成器**模拟 GPU 误差。例如：
- 用 PRNG（基于 seed）生成全图低频柏林噪声，模拟芯片级误差。
- 根据目标 GPU 的已知偏差模式（如 Intel 核显边缘偏暗），在边缘区域应用特定的卷积核。
这种方式更简单，但难以完全匹配真实 GPU 的高阶统计特征，适合初步方案。

---

## 总结

本方案通过**修改 Chromium 源码 + 嵌入式生成模型**，实现了高度隐蔽且确定性的 Canvas 指纹伪装。其关键优势在于：
- 噪声完全在 C++ 层生成，JS 无迹可寻。
- 模型学习真实 GPU 误差，使伪指纹具有统计自然性。
- 基于 profileID 的确定性保证了跨会话稳定性，同时又能为不同身份生成不同指纹。
