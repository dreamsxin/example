# Chrome 模拟方案

核心策略是：**保留硬件层的真实指纹，仅移除软件层的个性化特征，并通过确定性噪声打破 Canvas 追踪，同时严格保持所有信号的自洽**。

- 使用高斯噪声：基于 Box-Muller 变换，将均匀随机数转换为高斯分布。指定标准差 σ ≈ 0.8，这样大部分噪声集中在 -1 到 +1 之间，极少数情况出现 ±2。

- 上下文感知的噪声密度：分析 Canvas 命令流。如果检测到 createLinearGradient 或 shadowBlur，增加该区域的噪声概率和幅度；如果是纯色 fillRect，降低噪声概率甚至完全跳过。这需要解析 Canvas 绘制命令队列。

- 噪声频率与 GPU 型号绑定：不同架构的 GPU，其“高频抖动”和“低频色带”的表现不同。你可以预先针对你的 Intel UHD 620 采集 500 次 Canvas 渲染样本，训练一个小型生成模型（GAN 或 VAE），然后根据 profileID 生成符合该 GPU 纹理的噪声模式。

## 1. 信号链自洽模型

你需要让浏览器暴露的所有属性都能对得上同一台“标准宿主机”：

| 指纹维度 | 策略 | 实现方式 |
|----------|------|----------|
| 操作系统/版本 | 保持真实 | User-Agent、`navigator.platform`、`navigator.userAgentData` 中的平台字段完全使用宿主机的值，不修改。 |
| GPU 型号/驱动 | 保持真实 | WebGL 的 `UNMASKED_VENDOR_WEBGL`、`UNMASKED_RENDERER_WEBGL` 以及 Canvas 渲染特征均基于真实 GPU，不做任何伪装。 |
| CPU/内存 | 保持真实 | `hardwareConcurrency`、`deviceMemory` 等不做修改（若宿主机属于常见配置，则天然融入群体）。 |
| 屏幕 | 轻微修正 | 分辨率可使用真实值减去任务栏高度（Windows 40px），以匹配普通窗口化浏览行为；`devicePixelRatio` 保持真实。 |
| 字体列表 | **减少（裁剪）** | 仅暴露操作系统自带的默认字体 + 极少量高普及率字体（如 Office 核心字体），隐藏所有专业软件、开发工具、游戏带来的独家字体。 |
| 浏览器插件 | 清空/固定 | 将 `navigator.plugins` 和 `navigator.mimeTypes` 重置为标准 Chrome 内置插件（PDF Viewer、Chrome PDF Plugin 等），移除任何第三方插件。 |
| 时区/语言 | 保持真实（匹配 IP） | 使用宿主机时区和首选语言，但需确保与实际出口 IP 地理定位一致。 |
| Canvas 指纹 | **确定性地改变** | 基于 `profileID` 注入极微小的亚像素噪声（仅扰动最低有效位），使得不同 profile 产生不同的 Canvas 哈希，但噪声模式必须符合真实 GPU 的误差分布，避免产生“伪造痕迹”。 |
| AudioContext | 同样注入确定性噪声 | 对 DynamicsCompressor 的输出乘以 `1 + ε(profileID)`（ε ≈ 1e-7），保持与 Canvas 一致的伪装身份。 |

只要所有信号均指向同一套“真实硬件 + 精简软件”的组合，反检测系统就会将其归为正常设备，而不会触发异常警报。

## 2. 字体裁剪：只减不增，且不破坏系统字体

字体是最容易泄露用户身份的信号之一，你的要求是保留系统默认字体，移除所有第三方软件带来的额外字体。

### 2.1 实现原理
- 拦截所有字体枚举 API：`document.fonts`（`FontFaceSet`）、`navigator.fonts.query()`（需用户授权）、CSS 字体探测（`document.fonts.check()`）。
- 在页面脚本执行前注入过滤逻辑，使暴露的字体列表仅包含 **Win10/Win11 默认字体** + 可选的高覆盖字体（如微软雅黑、宋体、Arial、Times New Roman 等），并完全隐藏其他字体。
- 不实际删除系统字体文件，因此浏览器底层仍能用被隐藏的字体进行渲染，但字体探测脚本会发现“该字体不存在”。这样，Canvas 绘制文字时若指定了被隐藏的字体，浏览器会使用回退字体，从而微妙地改变 Canvas 指纹，这也是一种合法的特征变动。

### 2.2 代码示例（Puppeteer/CDP 脚本注入）
```javascript
// 在页面创建文档前注入
await page.evaluateOnNewDocument(() => {
  // 允许的字体白名单
  const WHITELIST = new Set([
    'arial', 'times new roman', 'courier new', 'verdana', 'tahoma',
    'microsoft yahei', 'simsun', 'simhei', 'mingliu', 'gulim', 'malgun gothic'
  ]);

  // 拦截 FontFaceSet.prototype.has/check
  const origCheck = FontFaceSet.prototype.check;
  FontFaceSet.prototype.check = function(font, text) {
    const family = font.split(' ')[0].replace(/['"]/g, '').toLowerCase();
    if (!WHITELIST.has(family)) return false;
    return origCheck.call(this, font, text);
  };

  // 拦截 queryLocalFonts (如果启用)
  if (navigator.fonts && navigator.fonts.query) {
    const origQuery = navigator.fonts.query;
    navigator.fonts.query = async function() {
      const allFonts = await origQuery.call(this);
      return allFonts.filter(f => WHITELIST.has(f.family.toLowerCase()));
    };
  }
});
```

**注意**：此方法无法阻止通过 `<canvas>` 测量文字宽度的间接探测，但因为你已裁剪了字体列表，探测脚本会发现被隐藏字体“不可用”，从而不会使用它们进行指纹计算，这本身就改变了指纹且不产生矛盾。

## 3. Canvas 确定性扰动：符合真实 GPU 特征的亚像素噪声

你保留了真实 GPU，因此 Canvas 渲染结果与所有同型号 GPU 用户相同。为了打破唯一性，你需要注入噪声，但又不能引入“非该 GPU 特征”的错误模式。

### 3.1 噪声设计原则
- **确定性**：噪声 = f(profileID, 像素坐标)，同一 profile 下每次绘制结果完全一致。
- **微小性**：仅修改像素值的最低 1~2 位，幅度控制在 ±2 以内，人眼不可见。
- **符合 GPU 误差分布**：不同 GPU 型号在浮点精度、抗锯齿等方面有特定的误差特征。例如，你的真实 GPU 是 Intel UHD Graphics 620，那么它的 Canvas 噪声通常集中在边缘像素的 alpha 通道上，且数值变化遵循类似高斯分布的模式。你需要**采集该 GPU 的基线噪声**，然后让注入的噪声遵循相同的分布。

### 3.2 基线采集与建模
你可以使用之前我提供的采集工具，在同一台机器上多次运行同一 Canvas 测试（如绘制贝塞尔曲线），每次重启浏览器或使用不同的“干净 profile”，观察像素值的波动（真实波动很小，通常在 ±1 内）。记录下这些波动发生的像素位置和幅度分布，构建一个特征模型。

### 3.3 注入实现
利用 Canvas 劫持，在 `toDataURL` / `getImageData` 被调用时，按预存的特征分布修改像素：
```javascript
// 伪代码
function applyProfileNoise(imageData, profileSeed) {
  const rng = seedrandom(profileSeed);
  for (let i = 0; i < data.length; i += 4) {
    // 以极低概率修改该像素，模拟真实 GPU 的随机波动
    if (rng() < 0.002) { // 概率可调
      data[i]   = clamp(data[i]   + rngInt(rng, -1, 1));
      data[i+1] = clamp(data[i+1] + rngInt(rng, -1, 1));
      data[i+2] = clamp(data[i+2] + rngInt(rng, -1, 1));
    }
  }
}
```
**关键**：这个噪声必须与你的 GPU 型号在其他信号上保持一致。不能出现“Canvas 呈现 Intel 核显的低频噪声”而 WebGL 却报告为“NVIDIA 独立显卡”的矛盾。

## 4. 避免信号矛盾的集成检查清单

模拟完成后，必须逐项验证以下信号全部匹配，否则就是严重破绽：

- [ ] User-Agent 中的平台字段（Windows NT 10.0）与 `navigator.platform`（Win32）匹配。
- [ ] WebGL `UNMASKED_RENDERER_WEBGL` 字符串与实际 GPU 一致（例如 `ANGLE (Intel, Intel(R) UHD Graphics 620 Direct3D11 vs_5_0 ps_5_0)`）。
- [ ] Canvas 渲染哈希在同 profile 下稳定，且其噪声模式符合 Intel UHD 的特征（如果你用的是这台 GPU）。
- [ ] AudioContext 指纹与 Canvas 噪声使用相同的 profileID 种子，且波动幅度一致。
- [ ] 屏幕分辨率、色深、像素比与实际窗口化设置匹配，不能是 3840×2160 却搭配 96 DPI。
- [ ] 字体列表仅包含白名单内的字体，且其中不含有任何与当前 GPU/系统不协调的字体（例如 Mac 专用字体出现在 Windows 上）。
- [ ] 时区、语言、键盘布局与出口 IP 地理定位不冲突（可以用 `Intl.DateTimeFormat().resolvedOptions()` 验证）。

## 5. 高级实现：Chromium 源码级修改（可选）

如果想要完全避免 JS 注入可能带来的原型链篡改痕迹，你可以直接修改 Chromium 源码，在底层实现：

- **字体枚举过滤**：在 `third_party/blink/renderer/platform/fonts/` 中修改 `FontCache` 或 `FontEnumeration` 相关逻辑，使系统返回的字体列表只包含白名单。
- **Canvas 噪声注入**：在 `third_party/blink/renderer/platform/graphics/canvas/` 的 `ImageDataBuffer` 或 `CanvasResourceProvider` 处，根据 `profileID` 叠加噪声，这样连 JS 都无法感知到修改。

这种方法隐蔽性最高，但需要维护自己的 Chromium 分支，适合专业反检测浏览器开发。

## 6. 总结：最终效果

通过上述设置，Chrome 将呈现出一个“和宿主机一模一样的硬件，但软件环境极其干净、且 Canvas 指纹带着一丝独特的、但符合该硬件自然波动的噪声”的形象。这样的信号链既不会因伪装成另一款 GPU 而被反指纹系统标记为异常，又能通过字体裁剪和确定性噪声打破跨站点追踪，实现“一人千面但面面相似”的隐匿效果。
