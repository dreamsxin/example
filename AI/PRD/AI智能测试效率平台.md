## 1. 背景与问题陈述

### 1.1 问题陈述

测试组当前面临三类相互加剧的问题：

1. **测试过程缺乏统一支撑**：回归测试依赖手工执行和分散脚本，执行结果散落各处，难以集中统计与追踪对比；质量趋势和缺陷分布缺乏数据支撑，问题检测依赖个人经验。需要一套统一平台将测试资产、执行数据和经验知识集中管理——让质量可见、问题可溯、经验可复用，帮助测试团队系统化建设。
2. **测试知识资产流失**：业务规则、历史缺陷模式、环境排障经验散落在个人头脑、聊天记录和各类文档中。新人上手周期长（通常以"月"计），同类问题反复踩坑，资深成员离职即意味着知识断层。
3. **存量测试工具碎片化**：测试组成员已各自借助 AI 为所负责的业务开发了测试工具。这证明 AI 提效路径真实可行、团队具备能动性，但这些"私有小工具"标准不一（语言/框架/报告格式各异）、无法统一调度（各自本地手动运行）、资产不可见（他人不知道、重复造轮子）、安全无保障（可能硬编码凭据、无权限无审计），且随人员流动而流失。

### 1.2 不解决的成本

| 维度 | 现状代价 |
| :--- | :--- |
| 交付速度 | 回归测试占用迭代周期 30%+ 的时间，版本发布节奏受制 |
| 质量风险 | 回归覆盖不足，线上缺陷中相当比例为"历史已知问题复发" |
| 人力成本 | 资深测试 30%~40% 时间用于答疑、带教和重复排障 |
| 知识风险 | 关键业务测试逻辑仅有 1~2 人掌握，存在单点依赖 |
| 工具资产风险 | 存量工具重复建设、无法纳入统一回归统计、存在凭据泄露隐患、人走工具失 |

> **核心洞察**：三个问题互为因果——知识沉淀不足导致用例编写慢、失败分析慢；测试过程缺乏统一支撑导致经验难以系统沉淀；而存量工具若不被平台收编，平台覆盖率建设就要从零开始、与成员已有成果形成"两张皮"。因此平台定位是**一体化系统：收编存量 + 统一治理 + 增量智能**，而非另起炉灶的新工具。

## 2. 产品愿景与目标

### 2.1 产品愿景

建设一套 **AI 驱动的智能测试效率平台**：以**自动化测试平台**为"执行引擎"、以 **RAG 测试知识库**为"知识引擎"，通过 **AI 能力中枢**将两者打通，并以**分级接入机制收编成员工具存量**，形成数据飞轮——知识库让自动化更聪明，自动化执行让知识库更丰富。最终让测试从"成本中心"转变为"质量引擎"，让**团队建设成果可追溯、可量化、可积累**。

**收编三不原则**：不强制重写（分级适配接入，降低接入成本）、不没收（工具负责人制，保留 owner 与署名）、不纵容（安全扫描与标准契约是准入底线）。

### 2.2 业务目标（可衡量）

| 编号 | 目标 | 衡量方式 | 目标值 |
| :--- | :--- | :--- | :--- |
| G1 | 缩短回归测试周期 | 全量回归从触发到报告产出的时长 | 上线 6 个月内降低 **50%** |
| G2 | 提升回归自动化覆盖率 | 核心业务流程自动化用例覆盖比例 | 上线 6 个月内达到 **70%**（主要路径：存量工具接入而非从零编写） |
| G3 | 降低重复问题发生率 | 线上缺陷中"历史已知问题复发"占比 | 上线 6 个月内下降 **30%** |
| G4 | 缩短新人上手周期 | 新测试成员独立完成核心模块回归的时间 | 从约 4 周缩短至 **2 周** |
| G5 | 收编存量工具资产 | 盘点后的存量工具完成 L2（结果上报）及以上接入的比例 | 上线 3 个月内达到 **80%** |

### 2.3 用户目标（测试团队视角）

- 测试工程师：重复执行交给平台，失败原因有 AI 辅助定位，**已有工具低成本接入不重写**，建设成果被看见。
- 测试组长：用例资产、**工具资产**、执行数据、知识资产可统一管理，团队贡献与建设成果可追溯、可量化。
- 开发工程师：代码提交后自动获得测试反馈，缺陷报告附带根因分析建议。

## 3. 非目标（Non-Goals）

| 非目标 | 说明 |
| :--- | :--- |
| **不替代人工探索性测试** | 平台覆盖可结构化的回归与验收测试；探索性测试、用户体验测试仍由人主导，AI 仅作辅助。 |
| **不强制重写存量工具** | 存量工具按 L1-L3 分级接入（见模块 J），适配优先；仅对少数高频高价值工具经评估后重构为原生子用例，且须 owner 自愿。 |
| **不做通用脚本托管市场** | 接入范围限定测试域工具（测试执行、数据构造、环境运维类），不演变为全司通用脚本平台。 |
| **不自研大模型** | LLM 与 Embedding 模型均采用成熟商用/开源方案（DeepSeek、GPT、BGE-M3 等），平台只做工程化集成。 |
| **不自研测试框架与性能压测内核** | Web UI/API/移动端/性能测试分别集成 Playwright、Requests/Rest-Assured、Appium、JMeter，平台做调度与编排，不重造轮子。 |
| **V1 不做 GraphRAG / 多跳知识推理** | V1 采用"向量 + 全文"混合检索；知识图谱推理列入 V2 演进方向，但数据模型需预留实体字段。 |
| **不自建 CI/CD 与缺陷系统** | 通过集成层对接现有 Jenkins/GitLab CI、Jira、Git，平台不做流程管理本身。 |
| **不做全公司通用知识库** | 知识库边界限定为测试域（用例、缺陷、业务规则、环境文档、工具文档），避免与全司知识中台职能重叠。 |

## 4. 用户角色与用户故事

### 4.1 角色定义

| 角色 | 描述 | 核心诉求 |
| :--- | :--- | :--- |
| 测试工程师 | 一线用例设计、执行、缺陷跟进者，多为自有测试工具的开发者 | 少写重复脚本，失败有分析，问题有答案，已有工具低成本接入、成果被看见 |
| 测试组长 / 负责人 | 测试策略与资产管理 | 覆盖率可视、任务可调度、知识可沉淀、**工具资产可盘点可收编** |
| 开发工程师 | 提交代码并接收质量反馈 | 提交后自动触发测试，失败报告能看懂 |
| 平台管理员（测试架构师） | 平台运维、环境、权限 | 环境稳定、资源可伸缩、敏感信息安全、**接入工具有安全底线** |
| 管理层 | 关注质量与效率趋势 | 一页纸看清质量大盘 |

### 4.2 用户故事（按优先级排序）

**P0**

1. 作为**测试工程师**，我希望在平台上按模块管理自动化用例（创建/编辑/版本/标签），以便用例资产有序沉淀、可按标签批量执行。
2. 作为**测试工程师**，我希望将我现有的测试工具登记到平台注册中心并补充使用文档，以便团队成员发现和使用，避免重复建设。
3. 作为**测试工程师**，我希望我的工具**保持现有运行方式**、仅通过 SDK 把执行结果上报平台，以便以最低成本纳入统一管理而不重写代码。
4. 作为**测试工程师**，我希望手动或定时触发一组用例执行并实时查看进度，以便灵活安排回归。
5. 作为**测试工程师**，我希望执行完成后收到包含通过率、失败截图、错误堆栈的报告，以便快速定位问题。
6. 作为**测试工程师**，我希望用自然语言提问（如"订单超时重试的边界条件有哪些历史缺陷？"）并获得带来源引用的答案，以便快速复用历史经验。
7. 作为**测试组长**，我希望把测试文档（需求、历史缺陷、环境手册、工具文档）批量导入知识库并自动完成清洗入库，以便知识集中管理。
8. 作为**平台管理员**，我希望所有接入工具通过敏感信息扫描（硬编码密码/Token 检测）且代码托管于 Git，以便消除存量工具的安全隐患。

**P1**

9. 作为**测试工程师**，我希望粘贴需求描述后由 AI 生成测试用例草稿（关联知识库中的相似历史用例），以便将用例设计时间缩短一半。
10. 作为**测试工程师**，我希望用例失败时平台自动检索相似历史缺陷与解决方案并给出根因建议，以便排障时间从小时级降到分钟级。
11. 作为**测试组长**，我希望高价值存量工具容器化改造后纳入平台统一调度，以便纳入全量回归并获得并行执行能力。
12. 作为**开发工程师**，我希望代码合并后自动触发关联模块的冒烟测试并将结果通知到群，以便尽早发现问题。
13. 作为**测试组长**，我希望测试执行产生的缺陷模式与修复方案经审核后回流知识库，以便知识库持续生长。
14. 作为**平台管理员**，我希望按角色配置权限并对 Token/密码等敏感配置加密存储，以便满足安全合规要求。

**P2（边界情况与体验）**

15. 作为**测试工程师**，当知识库没有匹配答案时，我希望系统明确告知"未命中"而非编造答案，以便我能判断信息可信度。
16. 作为**测试工程师**，当执行环境资源不足时，我希望任务排队并告知预计等待时间，以便安排工作。
17. 作为**测试工程师**，我希望 AI 生成的用例草稿标注"AI 生成，待人工评审"状态，以便与正式用例区分管理。
18. 作为**测试工程师**，我希望看到我的工具被团队调用的次数与贡献统计，以便我的建设成果被看见。

### 5.2 数据飞轮：双引擎如何互相增强

| 流向 | 内容 | 价值 |
| :--- | :--- | :--- |
| 知识库 → 自动化 | 用例生成时检索业务规则与相似历史用例作为上下文 | 生成更贴合业务、覆盖率更高的用例草稿 |
| 知识库 → 失败分析 | 失败日志检索相似历史缺陷与排障方案 | 分钟级给出根因建议 |
| 自动化 → 知识库 | 执行记录、缺陷模式、修复方案经审核后入库 | 知识库随每次执行持续生长，越用越聪明 |
| 工具 → 知识库 | 工具使用文档、FAQ、接入示例入库 | 新人可发现并复用他人工具，知识检索覆盖工具域 |
| 工具 → 报告体系 | L2 上报的执行结果纳入统一报告与覆盖率统计 | 覆盖率口径包含存量工具，G2 目标可达 |

## 6. 功能需求

### 模块 A：测试用例管理（P0）

**需求描述**：支持用例的创建、编辑、版本控制、分类、标签与关联管理；用例与测试数据分离；与需求、缺陷双向追溯。

**验收标准**：
- [ ] 用例支持创建/编辑/删除/复制，每次修改自动生成版本快照，可回滚至任意历史版本
- [ ] 用例支持多级目录分类与多标签标记，可按标签组合筛选并一键加入执行任务
- [ ] 用例可关联需求 ID（Jira）与缺陷 ID，从用例可跳转查看关联缺陷，从缺陷可反查关联用例
- [ ] 用例步骤中的测试数据以参数引用，修改数据不改用例本体
- [ ] AI 生成的用例草稿带有独立状态标记，经人工评审后方可转为正式用例

### 模块 B：测试任务调度（P0）

**需求描述**：测试任务的创建、优先级排序、分发与监控；支持手动、定时、事件三种触发方式；分布式调度。

**验收标准**：
- [ ] 支持三种触发方式：手动触发、Cron 定时触发（如每日构建后）、事件触发（Git 提交 / CI 流水线回调）
- [ ] 任务支持优先级设置，高优先级任务抢占空闲 Worker 资源
- [ ] 任务执行过程可实时查看进度（已完成/总数、当前用例、Worker 分布）
- [ ] 资源不足时任务进入队列，页面展示排队位置与预计等待时间
- [ ] 任务超时（可配置阈值）自动终止并标记异常
- [ ] 任务可混合编排"平台原生子用例"与"L3 接入的存量工具"为同一执行计划

### 模块 C：自动化测试执行引擎（P0）

**需求描述**：Master-Worker 分布式架构，驱动多类型测试框架执行，具备异常处理与重试机制。

**验收标准**：
- [ ] 支持 Web UI（Playwright）、API（Requests/Rest-Assured）两类测试的执行（V1 范围）
- [ ] 支持并行执行：单任务可拆分至多个 Worker 并行跑，全量回归耗时可随 Worker 数线性下降
- [ ] 单用例失败支持自动重试（次数可配），重试后仍失败才标记为失败，并保留全部尝试日志
- [ ] Worker 以 Docker 容器运行，环境一致性由镜像保障；Worker 异常掉线时其任务自动重新分配
- [ ] 执行过程采集完整证据链：步骤日志、截图（UI 类）、请求响应报文（API 类）

### 模块 D：测试数据管理（P1）

**需求描述**：测试数据的生成、隔离、版本控制与重置。

**验收标准**：
- [ ] 支持基于规则/Faker 的动态数据生成，配置数据与用例代码分离
- [ ] 支持按测试任务进行数据隔离（独立数据集或数据库快照恢复），并行任务互不干扰
- [ ] 敏感生产数据须经脱敏后方可进入测试数据池，脱敏规则可配置

### 模块 E：报告与可视化（P0）

**需求描述**：多维度测试报告与质量趋势分析，集成通知机制。

**验收标准**：
- [ ] 单次执行报告包含：通过率、失败用例清单、失败原因分类、证据链（截图/日志/报文）
- [ ] 报告统一涵盖"平台原生子用例执行"与"存量工具 L2/L3 上报"两类结果，覆盖率统计口径一致
- [ ] 报告可导出 HTML/PDF，可通过邮件、钉钉/企微机器人推送
- [ ] 质量大盘展示趋势：按周/月的通过率、自动化覆盖率、缺陷分布、flaky 用例 Top 榜、工具调用排行
- [ ] 管理层视图支持一页纸摘要（本周期质量结论 + 风险项）

### 模块 F：权限与安全（P0）

**验收标准**：
- [ ] 按角色（管理员/测试组长/测试工程师/只读）分配功能权限与数据权限
- [ ] 密码、Token、数据库凭据等敏感配置加密存储（落库加密），界面展示脱敏
- [ ] 平台操作日志（用例变更、任务触发、知识库增删、工具注册与接入变更）完整留痕可审计

### 模块 G：知识库离线数据管道（P0）

**需求描述**：六步流水线——**数据采集 → 格式转换与清洗 → 元数据标注 → 智能分块 → Embedding 向量化 → 向量数据库入库**。

**验收标准**：
- [ ] 支持 PDF、Word、Markdown、HTML、Excel 五类格式解析入库；结构化数据（数据库表、API）以连接器方式接入
- [ ] 清洗环节自动去除页眉页脚、页码、乱码，统一格式并去重
- [ ] 每条知识自动携带元数据：来源、作者、时间戳、密级、信源可信等级（T1 权威事实 ~ T4 未验证经验）
- [ ] 智能分块：按语义边界切分，块大小 500~1000 字，相邻块保留重叠窗口避免语义断裂
- [ ] 支持增量更新：文档变更仅重处理变更部分，不全量重建
- [ ] 入库前经检索质量验证（标准问题集抽检），通过率低于阈值则阻断发布并告警
- [ ] 工具注册中心的使用文档自动进入离线管道，无需二次手工导入

### 模块 H：知识检索与智能问答（P0）

**验收标准**：
- [ ] 问答采用混合检索（向量 + 全文）+ 重排序；答案必须附带来源引用，可点击跳转原文片段
- [ ] 未命中知识库时明确回答"知识库暂无相关内容"，**禁止编造**；高置信度与低置信度答案有视觉区分
- [ ] 支持多轮对话与追问；回答后可一键反馈"有用/无用"，反馈数据进入优化集
- [ ] 知识按密级做权限隔离，越权检索不可见
- [ ] 时效性知识（如环境地址、版本文档）带时间戳，过期内容自动降权并提示
- [ ] 工具类问题（"有没有测试 XX 的工具？"）可检索注册中心元数据并返回工具名、负责人与文档链接

### 模块 I：AI 智能测试能力（P1）

**I-1 用例智能生成**
- [ ] 输入需求描述/需求文档，AI 结合知识库检索结果（业务规则 + 相似历史用例）生成用例草稿
- [ ] 草稿默认"待评审"状态，须人工确认后转正式用例；记录采纳率用于效果评估
- [ ] 支持按指定覆盖率维度生成（如：边界值、异常流、权限场景）

**I-2 失败根因分析**
- [ ] 用例失败时自动触发：提取错误日志 → 检索相似历史缺陷与解决方案 → 生成根因建议与推荐操作
- [ ] 分析结论附在报告中，标注置信度；分析结果可被人工修正，修正记录回流知识库

**I-3 知识回流管道**
- [ ] 已闭环缺陷的"现象-根因-方案"可一键提交知识候选，经测试组长审核后入库
- [ ] 回流知识与原始缺陷单建立双向链接

### 模块 J：存量工具接入与治理（P0）

**需求描述**：将成员已各自开发的业务测试工具收编进平台，按"登记 → 上报 → 调度"分级接入，在**不重写**的前提下实现资产可见、数据统一、调度可行；同时建立安全准入与收编优先级机制。

**J-1 工具注册中心（P0）**
- [ ] 工具登记元数据：名称、业务域、类型（UI/API/数据构造/环境工具/其他）、负责人、代码仓库地址、使用文档、当前接入等级
- [ ] 工具列表支持按业务域/类型/负责人检索；疑似重复功能的工具自动提示，引导合并或复用
- [ ] 每个工具有且仅有一名负责人（owner），人员变动时须交接并变更登记

**J-2 分级接入模型（P0 含 L1/L2，P1 含 L3）**

| 等级 | 名称 | 工具侧改造 | 平台获得能力 | 参考成本 |
| :--- | :--- | :--- | :--- | :--- |
| L1 | 登记接入 | 无（仅注册元数据+文档） | 资产可见、文档可检索 | ≤0.5 人日 |
| L2 | 结果接入 | 引入上报 SDK，执行后上报结果 | 结果纳入统一报告与覆盖率统计 | 1~2 人日 |
| L3 | 调度接入 | 打包 Docker 镜像，遵循容器契约 | 纳入平台统一调度，可编排、可并行 | 3~5 人日 |
| L4 | 原生重构（可选） | 重构为平台原生子用例 | 完全纳管，享受全部平台能力 | 按评估，须 owner 自愿 |

- [ ] L2：平台提供 Python/Java 上报 SDK 与 API（字段：工具名、版本、业务域、结果、耗时、失败详情、证据链接）；上报失败可重试且不阻塞工具自身运行
- [ ] L3：定义《容器接入契约 v1》——镜像规范、环境变量入参、退出码语义、结构化结果输出（JSON 到约定路径/stdout）；不合规容器无法注册为可调度工具
- [ ] 平台提供接入 SDK、参考模板（含 Dockerfile 示例）与《工具接入规范 v1》文档
- [ ] 每个工具的执行记录均带工具版本号，可追溯"哪个版本产生了哪次结果"

**J-3 准入与安全治理（P0）**
- [ ] 准入三条件：代码托管于公司 Git、通过敏感信息扫描（硬编码密码/Token/内网地址检测）、指定负责人；缺一不予上架
- [ ] L3 镜像须通过基础安全扫描（漏洞基线）后方可被调度
- [ ] 工具申请访问平台敏感配置（如数据库凭据）时，须改用平台加密的配置中心，禁止自带明文
- [ ] 工具下线须保留历史执行数据与文档存档

**J-4 收编优先级与激励（P1）**
- [ ] 按四维度评分生成 L3 收编优先级清单：使用频率 × 业务关键度 × 稳定性（历史成功率）× 维护成本
- [ ] 高优先级工具可申请平台工程资源协助容器化改造
- [ ] 质量大盘展示"工具调用排行"与个人贡献统计，让建设成果可见

## 7. 技术方案概要

### 7.1 技术选型参考

| 领域 | 选型 | 说明 |
| :--- | :--- | :--- |
| 前端 | Vue.js | 控制台界面 |
| 后端服务 | Flask（Python）或 Spring Boot（Java） | 按团队技术栈定，倾向 Python（AI 生态一致，且与成员 AI 工具主流语言一致） |
| 任务调度 | Celery + 消息队列 | 分布式任务分发 |
| 工具接入 | 上报 SDK（Python/Java）+ Docker 容器契约 | 兼容存量工具多语言现状，L2 轻量、L3 标准化 |
| 安全扫描 | 敏感信息扫描（如 gitleaks 类）+ 镜像漏洞扫描 | 工具准入底线 |
| UI 自动化 | Playwright | 多浏览器、API 现代、稳定性优于 Selenium |
| API 自动化 | Requests / Rest-Assured | 视后端语言定 |
| 性能测试 | JMeter（集成） | V1.1 接入 |
| LLM | DeepSeek / GPT（可切换） | 用例生成、问答、失败分析 |
| Embedding | BGE-M3 或同级开源模型 | 中文场景语义匹配 |
| 向量数据库 | Milvus / FAISS / DashVector | 按数据规模与运维能力定 |
| 文档解析 | unstructured + PyMuPDF + python-docx | 多格式解析 |
| 编排框架 | LangChain / LlamaIndex | RAG 管道搭建 |
| 容器化 | Docker + Kubernetes | 执行环境一致性与弹性伸缩 |
| CI/CD 集成 | Jenkins / GitLab CI / GitHub Actions | 事件触发与流水线回调 |

### 7.2 关键架构原则

1. **解耦与标准化**：各层职责单一，通过标准接口通信；采集与处理经消息队列解耦。
2. **离在线分离**：知识准备（离线管道，定时/事件触发）与问答服务（在线管道，实时请求）独立部署、独立扩缩容。
3. **事件驱动**：数据源变更（如文档上传、缺陷关闭、工具注册）经事件总线自动触发处理管道，知识库近实时吸收新信息。
4. **AI 可兜底**：所有 AI 输出（用例草稿、根因建议、问答答案）均有人工评审/确认闭环，AI 不直接写生产资产。
5. **存量包容**：接入契约适配存量工具现状（多语言、多形态），平台不规定工具内部实现，只约定边界协议（上报协议、容器契约）。

## 9. 风险与合规

| 风险 | 等级 | 应对 |
| :--- | :--- | :--- |
| **AI 幻觉**：问答或用例生成编造不存在的内容 | 高 | 答案强制来源引用 + 未命中明确拒答 + 全部 AI 产出人工评审闭环；问答满意率纳入健康监控 |
| **敏感数据泄露**：需求文档、缺陷单含业务敏感信息进入向量库或发往外部 LLM | 高 | 知识密级分级 + 检索权限隔离；涉密内容仅走私有化部署模型；发送 LLM 的上下文经脱敏过滤；符合《个人信息保护法》要求，缺陷单中的用户信息入库前脱敏 |
| **存量工具安全隐患**：个人开发工具可能硬编码凭据、越权访问、无审计 | 高 | 准入三条件强制（Git 托管 + 敏感信息扫描 + 负责人制）；敏感配置统一收归平台加密配置中心；接入行为全量审计 |
| **收编引发抵触**：成员担心工具被"没收"或增加无谓工作量 | 中 | 三不原则（不重写/不没收/不纵容）写入制度；L1/L2 极低成本起步；工具调用排行与个人贡献公开展示形成正向激励；L3/L4 尊重 owner 意愿 |
| **测试数据合规**：生产数据用于测试 | 高 | 生产数据脱敏后方可入测试数据池，脱敏规则可审计 |
| **flaky 用例侵蚀信任**：不稳定的用例/工具导致报告失真、团队弃用平台 | 中 | 自动识别 flaky 项并隔离公示；重试机制 + 环境一致性容器化 |
| **LLM 成本失控**：用例生成与失败分析调用量随规模增长 | 中 | 调用量配额管理 + 小模型分级（简单问答走小模型）+ 成本月度复盘 |
| **知识过时**：环境文档等时效性知识过期误导使用者 | 中 | 时间戳管理 + 过期自动降权 + 定期巡检机制 |
| **范围蔓延**：被寄予"全司知识中台"或"全自动测试"期望 | 中 | 本 PRD 第 3 节非目标已明确边界，新增范围须走变更评审 |

## 11. 里程碑与分期建议

| 阶段 | 周期 | 范围 | 出口标准 |
| :--- | :--- | :--- | :--- |
| **MVP（V1.0）** | 第 1~3 个月 | 模块 A/B/C（API+Web UI）、E、F、G、H；**模块 J 的 J-1 注册中心 + J-2 的 L1/L2 接入 + J-3 准入治理** | 核心链路跑通：用例管理→调度执行→报告；知识库导入 3 类文档并开放问答；**存量工具完成盘点且 ≥80% 完成 L2 接入**；一个试点项目组全量使用 |
| **V1.1** | 第 4~5 个月 | 模块 D、I（用例生成 + 失败分析 + 知识回流）、**J-2 的 L3 容器调度接入（高优先级工具）+ J-4**、JMeter 接入 | AI 能力上线并产生采纳率数据；高价值存量工具纳入统一调度；数据飞轮闭环 |
| **V1.2** | 第 6 个月 | CI/CD 深度集成、质量大盘、flaky 治理 | 达到 G1~G5 阶段目标，输出复盘报告 |
| **V2.0（演进方向）** | 后续 | Appium 移动端、Serverless 弹性执行、GraphRAG 多跳推理、按领域拆分微型知识库、高价值工具 L4 原生重构 | 视 V1 复盘结论立项 |

> 原则：先用 MVP 解决最核心的三件事（**统一测试管理与问题统计** + 知识问答 + **存量工具收编可见**），跑通飞轮再叠加 AI 生成能力，**不追求一步到位的"完美平台"**。存量工具的 L2 接入是 G2 覆盖率目标最快可达的路径，须与平台主体建设并行启动，而非等平台完工再收编。

## 12. 附录：RAG 知识库测试环境验证清单

平台自身上线前，测试环境需优先验证以下关键节点：

1. 多格式文档解析的准确性（PDF 表格、Word 图片说明等边界场景）
2. 分块策略对检索效果的影响（块大小、重叠比例的对照实验）
3. Embedding 模型选型的语义匹配效果（中文业务术语召回率）
4. 向量数据库的检索性能（数据量 10 万/100 万 chunk 下的延迟）
5. 增量更新的正确性（文档变更后旧知识是否正确失效）
