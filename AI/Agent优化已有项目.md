**通用型「AI Agent 优化已有项目」模板**
----------------------------------------
它不绑定语言、不绑定框架，适合：

* 已有项目接入 AI 辅助开发
* 让 AI 做重构、补测试、补文档、优化流程
* 约束 AI 不乱改、不脑补、不大重写

你可以直接拆成两个文件使用：

* `README.md`：项目级说明
* `docs/ai-rules.md`：AI 执行硬约束

---

# 一、通用 `README.md` 模板

````md
# Project Name

> 一个持续演进中的项目。  
> 本项目已接入 AI Agent 辅助开发，目标是在**不破坏现有行为与稳定性**的前提下，逐步提升代码质量、可维护性、可测试性与文档完整度。

## 1. 项目背景

本项目是一个已存在的工程，当前重点不是从零开发，而是对既有系统进行持续优化，包括但不限于：

- 结构治理
- 历史模块整理
- 测试补齐
- 文档完善
- 错误处理统一
- 配置治理
- 日志与可观测性增强
- 性能优化
- AI Agent 辅助改造

---

## 2. 当前目标

当前阶段的目标：

- 保持业务稳定
- 降低维护成本
- 提升可读性与可测试性
- 降低模块耦合
- 建立 AI Agent 可安全参与的边界

非目标：

- 一次性推翻重写
- 未经验证的大规模替换
- 为“形式上的优雅”牺牲稳定性
- 在缺乏理解的前提下改动核心链路

---

## 3. AI Agent 参与原则

本项目允许 AI Agent 参与分析、重构、补测试、补文档和局部优化，但必须遵守以下原则。

### 3.1 总原则

AI 的工作优先级如下：

1. 不破坏现有行为
2. 先理解，再修改
3. 先小步修改，再逐步扩展
4. 先补保护，再做重构
5. 优先局部治理，避免整体重写

### 3.2 禁止行为

AI Agent 禁止：

- 未经说明直接大规模重构
- 擅自修改公共接口
- 擅自删除历史逻辑
- 假设不存在的文件、模块或依赖
- 引入未经批准的新依赖
- 用伪代码、占位符或 TODO 代替实现
- 为了“看起来更整洁”而改变既有行为
- 跳过错误处理、边界处理和兼容性评估

### 3.3 每次输出必须包含

AI 每次提交修改方案时，必须明确给出：

1. 修改目标
2. 修改原因
3. 涉及文件
4. 影响范围
5. 风险说明
6. 回滚建议（如适用）
7. 完整文件或可直接应用的 patch

---

## 4. AI Agent 可参与的工作范围

AI Agent 可以参与：

- 理解模块职责
- 梳理依赖关系
- 发现重复逻辑
- 提取公共逻辑
- 拆分超长函数
- 补充错误处理
- 补充日志
- 增加测试
- 增加文档
- 优化配置结构
- 清理明显死代码
- 做有依据的性能优化
- 输出渐进式重构方案

AI Agent 不应直接执行：

- 未审查的核心链路重写
- 高风险配置变更
- 未验证的数据结构迁移
- 未说明兼容性的接口变更
- 未验证的基础设施替换
- 默认扩大权限或关闭安全限制

## 5. 推荐目录原则

如果项目结构尚不清晰，可逐步向以下目标收敛：

```text
.
├── src/ / app/ / internal/         # 核心业务代码
├── modules/ / services/            # 业务模块或服务层
├── domain/                         # 领域模型与规则
├── adapters/ / integrations/       # 外部系统适配
├── api/ / interfaces/              # 对外接口定义
├── configs/                        # 配置模板与样例
├── scripts/                        # 构建、检查、发布脚本
├── tests/                          # 测试与测试资源
├── docs/                           # 文档
├── README.md
└── ...
````

说明：

* 新代码应优先进入清晰的业务层或模块层
* 不允许继续将复杂逻辑堆积在入口层或接口层
* 目录迁移应渐进式完成，不要求一次到位

## 6. 分层约束

### 6.1 接口层 / 入口层

职责：

* 接收输入
* 做基本校验
* 调用业务层
* 返回结果

禁止：

* 堆积复杂业务逻辑
* 直接操作存储细节
* 混入跨模块编排逻辑

### 6.2 业务层

职责：

* 承载业务流程
* 编排依赖调用
* 处理业务规则
* 组织事务边界或步骤顺序

禁止：

* 与具体展示层强耦合
* 把基础设施细节散落在各处
* 写成超长、不可测试的方法

### 6.3 数据访问 / 外部依赖层

职责：

* 数据访问
* 外部服务调用
* 对底层能力做统一封装

禁止：

* 承载业务规则
* 返回语义不清的错误
* 复制粘贴大量相似实现

### 6.4 领域层（如适用）

职责：

* 表达核心对象与规则
* 承载与外部协议无关的逻辑

禁止：

* 依赖接口层、展示层、底层框架细节

---

## 7. 重构策略

对已有项目进行优化时，必须遵循以下顺序：

### 第一步：理解现状

* 梳理模块职责
* 标出核心调用链
* 标记高风险路径
* 找出脏代码与重复逻辑

### 第二步：建立保护

* 补测试
* 加日志
* 补文档
* 明确输入输出行为

### 第三步：局部治理

* 先拆小函数
* 先提取重复逻辑
* 先统一错误处理
* 先收敛硬编码和魔法值

### 第四步：逐步收敛结构

* 调整目录
* 收敛依赖方向
* 拆分模块职责
* 提高可维护性

### 第五步：性能与可观测性优化

* 指标监控
* profile/trace 分析
* I/O 优化
* 并发控制
* 缓存与批处理策略

---

## 8. 代码规范

### 8.1 通用规范

* 代码必须可运行
* 保持单一职责
* 避免超长函数和超大文件
* 尽量避免循环依赖
* 新代码优先复用现有模块
* 尽量保持接口兼容

### 8.2 错误处理规范

* 不忽略错误
* 不用沉默失败掩盖问题
* 错误信息必须包含必要上下文
* 用户可见错误与内部错误应分离
* 不允许通过关闭校验或吞错来“解决问题”

### 8.3 日志规范

* 关键路径必须有日志
* 错误日志必须带上下文
* 禁止记录敏感信息
* 日志应便于搜索和排查

### 8.4 配置规范

* 禁止硬编码环境差异
* 所有配置集中管理
* 敏感配置不进入代码仓库
* 配置项应有默认值、说明或校验

---

## 9. 测试要求

### 最低要求

* 新增逻辑必须有测试
* 修复 bug 必须补回归测试
* 重构前应优先补保护性测试

### 测试类型

* 单元测试：核心逻辑、工具函数、规则判断
* 集成测试：模块协作、存储、外部依赖
* 回归测试：历史问题复现与防回归
* 基准测试：热点路径性能验证（如适用）

### 禁止

* 只写形式测试
* mock 过度导致测试失真
* 不验证关键业务结果
* 用“测试通过”代替行为分析

---

## 10. 性能优化约束

AI Agent 可以做性能优化，但必须：

1. 先说明瓶颈判断依据
2. 说明优化假设
3. 明确影响范围
4. 优先给出可验证依据
5. 不以可读性彻底崩坏为代价

常见方向：

* 减少重复计算
* 优化热点路径
* 控制并发开销
* 批量化 I/O
* 减少序列化损耗
* 优化查询与缓存策略

---

## 11. 兼容性原则

任何修改都必须评估以下兼容性风险：

* 是否影响对外接口
* 是否影响历史数据
* 是否影响配置项
* 是否影响日志与监控
* 是否影响上下游模块
* 是否影响已有脚本或部署流程

如有兼容性风险，必须显式说明。

## 12. 开发流程

### 分支策略

* `main`：稳定分支
* `dev`：开发集成分支
* `feature/*`：功能开发
* `fix/*`：问题修复
* `refactor/*`：重构治理

### 提交流程

1. 先理解目标模块
2. 再提出最小修改方案
3. 本地验证或补测试
4. 输出完整修改内容
5. 说明风险与影响
6. 审查通过后合并

### Commit 规范

```text
feat: 新增某功能
fix: 修复某问题
refactor: 重构某模块
docs: 更新文档
test: 增加测试
chore: 其他维护
perf: 性能优化
```

````
```md
# AI Agent Rules

本文件定义 AI Agent 在本项目中的行为边界。  
优先级高于普通开发建议。

---

## 1. 总原则

AI Agent 的职责是帮助项目做**渐进式优化**，不是无边界重写。

目标：

- 保持现有行为
- 降低维护成本
- 提升可测试性
- 提升可观测性
- 提高代码清晰度
- 提供可验证、可回滚的修改

---

## 2. 先分析，后修改

在修改任何已有代码前，AI 必须先做以下动作：

1. 理解该模块的输入输出
2. 确认调用方与被调用方
3. 判断是否属于核心链路
4. 判断是否已有测试覆盖
5. 标出潜在风险点

如果信息不足，优先给出保守修改方案，不进行激进重构。

---

## 3. 渐进式修改原则

允许的改法：

- 提取小函数
- 提取重复逻辑
- 统一错误处理
- 补充参数校验
- 增加日志
- 增加测试
- 小范围模块拆分
- 清理明显死代码

谨慎进行的改法：

- 调整目录结构
- 变更接口定义
- 替换并发模型
- 替换基础依赖
- 重写持久化层
- 改变事务边界或流程顺序

禁止的改法：

- 一次性大重写
- 未说明原因删除历史逻辑
- 未验证就改核心链路
- 未经批准新增大量依赖
- 通过吞错、跳过校验来换取“看起来正常”

---

## 4. 输出格式要求

每次输出修改方案时，必须包含：

### 4.1 修改说明
- 改了什么
- 为什么改
- 涉及哪些文件

### 4.2 风险说明
- 是否有行为变化风险
- 是否影响兼容性
- 是否影响性能
- 是否影响测试

### 4.3 交付格式
优先顺序：

1. 完整文件
2. 可直接应用的 patch
3. 分步骤提交建议

默认不要只输出片段，除非明确要求。

---

## 5. 通用工程约束

### 5.1 错误处理
- 不忽略错误
- 不允许沉默失败
- 不允许用占位实现掩盖问题
- 错误必须包含必要上下文

### 5.2 结构约束
- 新代码优先进入清晰模块边界
- 避免循环依赖
- 避免巨型工具文件或万能模块
- 不继续加重旧结构混乱

### 5.3 接口设计
- 优先最小变更
- 保持接口兼容
- 不为了抽象而抽象
- 不新增无意义包装层

### 5.4 并发与异步
- 必须考虑退出、超时、取消与失败
- 必须避免资源泄漏
- 共享状态必须有边界和同步策略

### 5.5 配置与环境
- 不硬编码敏感值
- 不混淆开发、测试、生产配置
- 所有新配置必须可说明、可校验

---

## 6. 测试规则

修改代码时必须判断是否要补测试。

### 必须补测试的情况
- 修 bug
- 重构核心逻辑
- 增加分支逻辑
- 调整错误处理
- 调整并发/异步逻辑
- 修改外部接口行为

### 测试要求
- 验证行为，不只是跑通
- 尽量覆盖边界条件
- 避免与实现细节强耦合
- 回归测试要能复现历史问题

---

## 7. 文档规则

以下修改应同步文档：

- 新配置项
- 新接口
- 目录结构调整
- 新运行方式
- 新限制条件
- 高风险模块变更

---

## 8. 性能优化规则

做性能优化前，至少应满足以下之一：

- 有 profile 证据
- 有 benchmark 数据
- 有明确热点路径
- 有线上指标反馈

禁止：

- 无依据进行“想当然优化”
- 为极小收益引入巨大复杂度
- 不解释就改关键并发或缓存逻辑

---

## 9. 安全规则

禁止：

- 输出真实密钥
- 建议关闭鉴权或校验以图方便
- 建议删除日志、审计或监控
- 默认扩大权限范围
- 让高风险操作默认自动执行

必须：

- 对敏感信息做脱敏
- 对高风险行为给出保护方案
- 对权限边界给出明确说明
- 对外部调用设置失败与超时策略

---

## 10. 与人类协作方式

AI Agent 不是最终裁决者。

对于以下情况，AI 应明确标记风险，而不是擅自决定：

- 公共接口改动
- 数据模型或存储结构变更
- 核心主流程重构
- 基础依赖替换
- 安全策略调整
- 大规模目录迁移

---

## 11. 默认工作顺序

遇到一个已有模块时，AI 默认按以下顺序工作：

1. 理解现状
2. 描述问题
3. 标记风险
4. 给出最小修改方案
5. 补测试
6. 再考虑进一步重构
7. 最后考虑性能优化

---

## 12. 一句话要求

> 小步修改、保持兼容、先测后改、先稳后优。
````

````md
3. 先建立保护，再进行重构
4. 先做局部治理，再逐步扩大
5. 可验证优先于“看起来更好”

### 3.2 工程原则

- 优先复用现有模块，避免重复实现
- 尽量保持接口兼容
- 所有新增逻辑必须可解释、可测试、可回滚
- 所有高风险改动必须显式标记风险
- 所有配置、依赖、边界行为必须可追踪

---

## 4. AI Agent 参与原则

本项目允许 AI Agent 参与分析、重构、补测试、补文档和局部优化，但必须遵守以下原则。

### 4.1 允许 AI Agent 做的事

- 理解模块职责
- 梳理依赖关系
- 发现重复逻辑
- 提取公共逻辑
- 拆分超长函数或模块
- 补充错误处理
- 补充日志
- 增加测试
- 增加文档
- 输出渐进式重构方案
- 做有依据的性能优化

### 4.2 禁止 AI Agent 做的事

- 未经说明直接大规模重构
- 擅自修改公共接口
- 擅自删除历史逻辑
- 假设不存在的文件、模块、依赖或上下文
- 引入未经批准的新依赖
- 用伪代码、占位符或 TODO 代替实现
- 通过吞错、跳过校验来“修复问题”
- 为“看起来更整洁”而改变既有行为
- 在没有验证的情况下顺手改很多地方

### 4.3 AI 每次输出必须包含

1. 修改目标
2. 修改原因
3. 涉及文件
4. 影响范围
5. 风险说明
````

## 6. 分层约束

### 6.2 业务层

* 承载业务流程
* 编排依赖调用
* 处理业务规则

禁止：

禁止：

### 6.4 领域层（如适用）

````
# AI Agent Rules

本文件定义 AI Agent 在本项目中的行为边界。  
优先级高于普通开发建议。

---

## 1. 总原则

AI Agent 的职责是帮助项目做**渐进式优化**，不是无边界重写。

目标：

- 保持现有行为
- 降低维护成本
- 提升可测试性
- 提升可观测性
- 提高代码清晰度
- 提供可验证、可回滚的修改

---

## 2. 先分析，后修改

在修改任何已有代码前，AI 必须先做以下动作：

1. 理解该模块的输入输出
2. 确认调用方与被调用方
3. 判断是否属于核心链路
4. 判断是否已有测试覆盖
5. 标出潜在风险点

如果信息不足，优先给出保守修改方案，不进行激进重构。

---

## 3. 渐进式修改原则

允许的改法：

- 提取小函数
- 提取重复逻辑
- 统一错误处理
- 补充参数校验
- 增加日志
- 增加测试
- 小范围模块拆分
- 清理明显死代码

谨慎进行的改法：

- 调整目录结构
- 变更接口定义
- 替换并发模型
- 替换基础依赖
- 重写持久化层
- 改变事务边界或流程顺序

禁止的改法：

- 一次性大重写
- 未说明原因删除历史逻辑
- 未验证就改核心链路
- 未经批准新增大量依赖
- 通过吞错、跳过校验来换取“看起来正常”

---

## 4. 输出格式要求

每次输出修改方案时，必须包含：

### 4.1 修改说明

- 改了什么
- 为什么改
- 涉及哪些文件

### 4.2 风险说明

- 是否有行为变化风险
- 是否影响兼容性
- 是否影响性能
- 是否影响测试

### 4.3 交付格式

优先顺序：

1. 完整文件
2. 可直接应用的 patch
3. 分步骤提交建议

默认不要只输出片段，除非明确要求。

---

## 5. 通用工程约束

### 5.1 错误处理

- 不忽略错误
- 不允许沉默失败
- 不允许用占位实现掩盖问题
- 错误必须包含必要上下文

### 5.2 结构约束

- 新代码优先进入清晰模块边界
- 避免循环依赖
- 避免巨型工具文件或万能模块
- 不继续加重旧结构混乱

### 5.3 接口设计

- 优先最小变更
- 保持接口兼容
- 不为了抽象而抽象
- 不新增无意义包装层

### 5.4 并发与异步

- 必须考虑退出、超时、取消与失败
- 必须避免资源泄漏
- 共享状态必须有边界和同步策略

### 5.5 配置与环境

- 不硬编码敏感值
- 不混淆开发、测试、生产配置
- 所有新配置必须可说明、可校验

---

## 6. 测试规则

修改代码时必须判断是否要补测试。

### 必须补测试的情况

- 修 bug
- 重构核心逻辑
- 增加分支逻辑
- 调整错误处理
- 调整并发或异步逻辑
- 修改外部接口行为

### 测试要求

- 验证行为，不只是跑通
- 尽量覆盖边界条件
- 避免与实现细节强耦合
- 回归测试要能复现历史问题

---

## 7. 文档规则

以下修改应同步文档：

- 新配置项
- 新接口
- 目录结构调整
- 新运行方式
- 新限制条件
- 高风险模块变更

---

## 8. 性能优化规则

做性能优化前，至少应满足以下之一：

- 有 profile 证据
- 有 benchmark 数据
- 有明确热点路径
- 有线上指标反馈

禁止：

- 无依据进行“想当然优化”
- 为极小收益引入巨大复杂度
- 不解释就改关键并发或缓存逻辑

---

## 9. 安全规则

禁止：

- 输出真实密钥
- 建议关闭鉴权或校验以图方便
- 建议删除日志、审计或监控
- 默认扩大权限范围
- 让高风险操作默认自动执行

必须：

- 对敏感信息做脱敏
- 对高风险行为给出保护方案
- 对权限边界给出明确说明
- 对外部调用设置失败与超时策略

---

## 10. 与人类协作方式

AI Agent 不是最终裁决者。

对于以下情况，AI 应明确标记风险，而不是擅自决定：

- 公共接口改动
- 数据模型或存储结构变更
- 核心主流程重构
- 基础依赖替换
- 安全策略调整
- 大规模目录迁移

---

## 11. 默认工作顺序

遇到一个已有模块时，AI 默认按以下顺序工作：

1. 理解现状
2. 描述问题
3. 标记风险
4. 给出最小修改方案
5. 补测试
6. 再考虑进一步重构
7. 最后考虑性能优化

---

## 12. 输出模板

每次 AI 修改建议都应尽量遵循以下格式：

### 修改目标
说明要解决什么问题。

### 修改原因
说明为什么需要这次修改。

### 涉及文件
列出会改动的文件。

### 风险评估
说明可能影响的行为、兼容性、性能或测试。

### 实施方案
描述采用的最小修改方案。

### 测试建议
列出应补充或执行的测试。

### 交付内容
默认给出完整文件；如不适合，给出可直接应用的 patch。

---

## 13. 一句话要求

> 小步修改、保持兼容、先测后改、先稳后优。
````

```md
# Refactor Plan

本文档用于指导已有项目的渐进式优化与重构。  
目标不是推翻重写，而是在尽量保持现有行为稳定的前提下，逐步改善结构、质量和可维护性。

---

## 1. 重构目标

本轮重构的目标通常包括：

- 明确模块职责
- 降低耦合度
- 提升可测试性
- 统一错误处理
- 提升日志与可观测性
- 减少重复逻辑
- 为后续功能迭代建立更清晰的边界

不在当前目标内的事项：

- 一次性重写全部历史代码
- 未验证的架构大迁移
- 没有收益证明的大规模技术替换

---

## 2. 重构原则

### 2.1 基本原则

- 保持现有行为尽量不变
- 优先做可验证的小步改动
- 先建立保护，再修改结构
- 先局部治理，再考虑全局收敛
- 高风险模块优先做文档和测试补强

### 2.2 失败标准

以下情况视为失败或高风险信号：

- 重构后行为与原逻辑不一致
- 修改范围失控
- 无法验证改动正确性
- 接口兼容性被破坏但未说明
- 引入新的隐性耦合或技术债

---

## 3. 重构范围模板

本轮重构关注范围：

- 模块：
- 文件：
- 关键调用链：
- 依赖边界：
- 当前主要问题：

本轮不包含：

- 模块：
- 文件：
- 高风险区域：
- 外部依赖迁移：
- 存储或协议变更：

---

## 4. 分阶段策略

### 阶段一：现状梳理

目标：

- 理解模块职责
- 梳理调用链
- 标记核心链路
- 找出明显重复逻辑与坏味道

交付物：

- 模块清单
- 风险点清单
- 重构优先级建议
- 待补测试点清单

完成标准：

- 说清楚当前逻辑如何工作
- 标出不能轻易动的核心部分
- 明确“先改哪里最安全”

---

### 阶段二：建立保护

目标：

- 补充保护性测试
- 增加必要日志
- 补齐关键文档
- 明确关键输入输出

交付物：

- 新增或补全的测试
- 关键路径日志点说明
- 行为说明文档

完成标准：

- 核心路径至少有基本行为保护
- 修改前后可以做最小验证
- 历史风险点有记录

---

### 阶段三：局部治理

目标：

- 拆分超长函数
- 提取重复逻辑
- 统一错误处理
- 收敛硬编码和魔法值
- 清理明显死代码

交付物：

- 小步提交清单
- 局部重构后的模块说明
- 新旧行为一致性说明

完成标准：

- 单次改动范围可控
- 改动后模块更清晰
- 不引入新行为偏差

---

### 阶段四：结构收敛

目标：

- 调整模块边界
- 优化依赖方向
- 逐步把逻辑从入口层/接口层迁出
- 收敛到更清晰的层次结构

交付物：

- 目录或模块调整方案
- 依赖方向说明
- 新结构迁移计划

完成标准：

- 新代码不继续加重旧结构混乱
- 核心模块职责更单一
- 依赖关系更容易理解和测试

---

### 阶段五：性能与可观测性优化

目标：

- 识别热点路径
- 做有依据的性能优化
- 增加指标、trace 或 profile 支撑
- 优化关键路径的资源消耗

交付物：

- 性能瓶颈说明
- 优化前后对比
- 观测指标说明

完成标准：

- 优化有证据支持
- 可读性和复杂度可接受
- 风险可评估

---

## 5. 重构优先级模型

建议按以下优先级排序处理模块：

### 优先级高

- 业务重要但可测试性差
- 高频变更且容易出错
- 重复逻辑多
- 错误处理混乱
- 高耦合且已拖慢开发效率

### 优先级中

- 结构混乱但业务风险一般
- 历史代码质量差但改动频率不高
- 测试缺失但影响面有限

### 优先级低

- 几乎不变更的边缘模块
- 稳定运行且无明显收益点的区域
- 暂时没有验证手段的高风险深层逻辑

---

## 6. 单次重构任务模板

每次重构任务应包含：

### 任务名称
例如：拆分用户服务中的状态校验与持久化逻辑

### 当前问题
描述当前模块存在什么问题。

### 目标
描述本次改动只想解决什么问题。

### 不做什么
明确这次不处理的范围，避免任务膨胀。

### 风险
说明可能影响哪些行为、接口、性能或兼容性。

### 保护措施
列出已有测试、拟补测试、日志、开关或回滚手段。

### 实施步骤
1.  
2.  
3.  

### 验证方法
- 单元测试
- 集成测试
- 手工验证
- 日志核查
- 指标对比

### 完成标准
说明什么情况下算完成。

---

## 7. 风险清单模板

重构前建议至少检查以下风险：

- 是否影响对外接口
- 是否影响历史数据
- 是否影响配置项
- 是否影响已有脚本或部署流程
- 是否影响日志和监控
- 是否影响性能或资源消耗
- 是否影响并发、异步或缓存行为
- 是否影响上下游依赖模块

---

## 8. 回滚策略模板

对中高风险改动，应提前准备回滚思路。

### 回滚方式

- 回退到上一个稳定提交
- 保留旧实现并通过开关切换
- 仅回滚新增加的模块边界调整
- 保留兼容层，延后彻底删除旧路径

### 回滚前提

- 新旧逻辑可切换
- 改动范围清晰
- 配置或数据变更可逆
- 验证路径可重复执行

---

## 9. 验收标准

一轮重构至少应满足：

- 当前行为没有被无意改变
- 关键路径有基本测试保护
- 模块职责更清晰
- 重复逻辑减少
- 风险说明明确
- 文档已同步更新
- 后续迭代比之前更容易

---

## 10. 推荐执行节奏

建议一个健康的重构节奏如下：

1. 先分析，不着急改
2. 先补保护，再动结构
3. 每次只做一个明确目标
4. 每次改动都能独立验证
5. 优先处理最阻碍开发效率的问题
6. 避免在单次任务里顺手“优化一切”

---

## 11. 一句话原则

> 把大问题拆成可验证的小修改，把高风险改动延后，把可回滚能力提前。
```

```md
# Testing Guide

本文档定义项目中的测试目标、测试层次、补测策略和 AI Agent 参与测试工作的约束。

---

## 1. 测试目标

测试的目标不是让 CI 变绿，而是确保：

- 关键行为可验证
- 重构有保护
- bug 修复可防回归
- 核心模块的输入输出清晰
- 风险修改有最基本的安全网

---

## 2. 测试原则

### 2.1 基本原则

- 优先测试行为，不测试无意义实现细节
- 优先覆盖高风险路径
- 修 bug 必须补回归测试
- 重构前优先补保护性测试
- 测试应尽量稳定、可重复、可读

### 2.2 禁止事项

- 只写形式测试
- 不断 mock 导致测试失真
- 测试名称无法表达意图
- 用测试绕过实际问题
- 测试依赖不稳定的外部环境而无隔离措施

---

## 3. 测试层次

### 3.1 单元测试

适用场景：

- 纯逻辑函数
- 规则判断
- 数据转换
- 小型服务方法
- 边界条件处理

目标：

- 覆盖核心分支
- 覆盖错误分支
- 覆盖边界条件

### 3.2 集成测试

适用场景：

- 模块间协作
- 数据访问
- 缓存
- 消息、队列、外部服务适配
- 复杂业务流程

目标：

- 验证关键协作链路
- 验证主要依赖组合后的行为

### 3.3 回归测试

适用场景：

- 历史 bug 修复
- 高风险逻辑修正
- 易反复出现的问题
- 曾经出现过线上事故的路径

目标：

- 明确记录问题场景
- 防止相同问题再次出现

### 3.4 基准测试 / 性能测试

适用场景：

- 热点路径
- 高并发逻辑
- 大数据量处理
- 已识别的性能瓶颈

目标：

- 有依据地验证优化收益
- 防止性能退化

---

## 4. 何时必须补测试

以下情况必须补测试：

- 修复 bug
- 增加新分支逻辑
- 重构核心逻辑
- 调整错误处理
- 修改并发或异步流程
- 修改对外接口行为
- 优化热点路径
- 修复边界条件问题

以下情况建议补测试：

- 拆分超长函数
- 提取公共逻辑
- 清理重复逻辑
- 收敛魔法值和硬编码
- 补日志或补文档时顺带澄清行为

---

## 5. 保护性测试策略

对已有项目做优化时，优先写“保护性测试”。

保护性测试的目的不是验证新架构，而是先记录当前行为，防止重构时无意改坏。

保护性测试通常应回答以下问题：

- 当前输入在什么条件下会得到什么输出
- 哪些错误分支会被触发
- 哪些边界条件必须保持
- 哪些副作用必须发生或不能发生

写保护性测试时：

- 先尊重当前行为
- 先覆盖关键场景
- 不急于追求完美覆盖率
- 不在第一次就重写全部测试

---

## 6. 测试设计建议

### 6.1 一个好测试应具备

- 名称清晰
- 前置条件明确
- 行为断言明确
- 与无关实现细节解耦
- 失败时容易定位原因

### 6.2 推荐结构

一个测试建议包含：

1. 场景说明
2. 输入准备
3. 行为执行
4. 结果断言
5. 必要的副作用断言

### 6.3 重点覆盖的内容

- 正常路径
- 错误路径
- 空输入
- 边界值
- 重复调用
- 并发/异步场景
- 超时/失败/取消场景
- 配置缺失或异常场景

---

## 7. Mock 与替身原则

可以使用 mock、stub、fake，但必须克制。

### 推荐使用场景

- 外部服务不可稳定依赖
- 需要隔离高成本依赖
- 需要精确控制返回值、异常或延迟
- 需要验证少量关键交互

### 不推荐使用场景

- 把所有依赖全部 mock 掉
- 测试只验证“函数被调用过”
- 用复杂 mock 替代真实行为验证
- 为了 mock 而过度重构生产代码

### 原则

- 优先验证结果，其次验证交互
- 优先简单替身，避免复杂 mock 脚本
- 能用小型真实依赖验证时，不必过度 mock

---

## 8. 集成测试建议

集成测试应重点覆盖：

- 关键业务流程
- 模块之间的协作关系
- 存储读写行为
- 外部系统调用边界
- 配置与初始化行为
- 错误传播路径

集成测试应尽量做到：

- 可重复执行
- 环境要求清晰
- 失败信息可定位
- 尽量减少对外部不稳定依赖

---

## 9. 回归测试模板

每个已修复问题建议记录：

### 问题名称
例如：重复提交导致状态异常更新

### 问题描述
说明历史 bug 的现象与触发条件。

### 复现条件
列出输入、状态、顺序或环境条件。

### 期望行为
说明修复后应当如何表现。

### 回归测试位置
记录对应测试文件或测试名称。

---

## 10. 性能测试建议

在以下情况下考虑增加性能测试：

- 代码位于热点路径
- 变更可能影响吞吐、延迟或资源消耗
- 需要验证优化是否有效
- 需要防止后续出现性能退化

性能测试建议至少记录：

- 测试目标
- 输入规模
- 指标维度
- 对比基线
- 结果结论

---

## 11. AI Agent 与测试协作规则

AI Agent 在补测试时必须遵守以下规则：

1. 先理解当前行为，再写测试
2. 修 bug 必须给出回归测试建议
3. 重构前优先补保护性测试
4. 测试必须验证行为，而不是只验证实现细节
5. 默认不要为了写测试而大改生产代码
6. 如果必须调整结构以便测试，应明确说明原因和范围
7. 不允许伪造测试结果
8. 不允许声称“已通过测试”除非已有明确执行结果

---

## 12. 测试完成标准

一项改动的测试工作至少应满足：

- 关键行为已覆盖
- 风险路径已有保护
- 历史 bug 已有回归测试
- 测试命名可读
- 测试能稳定重复运行
- 失败时可快速定位原因

---

## 13. 推荐测试检查清单

提交前建议检查：

- [ ] 是否覆盖正常路径
- [ ] 是否覆盖错误路径
- [ ] 是否覆盖边界条件
- [ ] 修 bug 是否补了回归测试
- [ ] 重构前是否补了保护性测试
- [ ] 是否有过度 mock
- [ ] 测试是否依赖脆弱环境
- [ ] 测试名是否能说明行为
- [ ] 失败信息是否足够清晰

---

## 14. 一句话原则

> 先用测试守住当前行为，再用重构改善代码质量。
```

````md
本文档描述项目当前的核心架构、模块边界、依赖方向、关键调用链与高风险区域。  
目标是帮助开发者与 AI Agent 在**不破坏现有行为**的前提下，对项目进行渐进式优化。

---

## 1. 文档目标

本文档用于回答以下问题：

- 这个项目现在是怎么工作的
- 核心模块分别负责什么
- 模块之间如何依赖
- 哪些调用链是关键路径
- 哪些区域风险高，不能轻易改
- AI Agent 可以优先优化哪些区域
- 后续架构应向什么方向收敛

本文档强调：

- **先描述现状**
- **再定义边界**
- **最后提出收敛方向**

本文档不要求系统一开始就是“完美架构”。

---

## 2. 架构原则

本项目遵循以下基础原则：

1. 保持现有行为稳定
2. 模块职责尽量单一
3. 接口层、业务层、数据层职责分离
4. 外部依赖通过边界适配，避免直接散落在业务代码中
5. 高风险逻辑优先保护，低风险逻辑优先治理
6. 架构优化必须渐进式完成，不做无边界重写

## 4. 逻辑分层

本项目推荐按如下逻辑分层理解，即使当前代码并未完全按此组织，也应以此为分析和收敛方向。

### 4.1 接口层 / 入口层

职责：

- 接收请求或输入
- 做参数校验
- 调用业务层
- 返回结果或触发流程

典型位置：

- `api/`
- `handler/`
- `controller/`
- `transport/`
- `cmd/`
- `cli/`

不应承担：

- 复杂业务规则
- 跨模块事务编排
- 底层存储细节
- 过多外部依赖处理

---

### 4.2 业务层

职责：

- 组织核心流程
- 编排模块调用
- 实现业务规则
- 管理执行顺序、步骤边界与事务边界

典型位置：

- `service/`
- `application/`
- `usecase/`
- `modules/*`

不应承担：

- 展示层细节
- 具体存储实现细节
- 与具体第三方 SDK 的强耦合调用

---

### 4.3 领域层（如适用）

职责：

- 表达领域对象
- 承载核心规则
- 封装与外部协议无关的业务逻辑

典型位置：

- `domain/`
- `model/`
- `entity/`

不应承担：

- HTTP / RPC / UI 细节
- ORM / SQL / SDK 细节
- 框架耦合逻辑

---

### 4.4 数据访问 / 外部依赖层

职责：

- 数据存取
- 第三方服务访问
- 缓存、消息、文件等底层能力封装

典型位置：

- `repository/`
- `dao/`
- `store/`
- `adapters/`
- `integrations/`

不应承担：

- 复杂业务判断
- 跨业务模块编排
- 业务语义不明确的错误封装
````

## 6. 依赖方向约束

推荐依赖方向如下：

```text
接口层 / 入口层
    ↓
业务层
    ↓
领域层（可选）
    ↓
数据访问 / 外部依赖层
```

额外说明：

* 接口层可以依赖业务层，但不应反向被业务层依赖
* 业务层可以调用数据访问层，但不应直接散落底层实现细节
* 领域层尽量不依赖具体基础设施
* 公共工具模块不应反向依赖业务模块
* 配置、日志、错误处理应作为基础能力被复用，而不是到处复制

### 不推荐的依赖关系

* 接口层直接拼接存储逻辑
* 业务层与 UI / HTTP / RPC 强耦合
* 数据层承载业务规则
* 工具层依赖具体业务模块
* 模块之间双向依赖或循环依赖

## 9. 配置架构

### 9.1 配置来源

系统配置可能来自：

* 本地配置文件
* 环境变量
* 启动参数
* 密钥管理系统
* 数据库存储
* 远程配置中心

### 9.2 配置原则

* 环境差异通过配置体现，不写死在代码中
* 敏感信息不得硬编码
* 配置项必须有说明、默认值或校验
* 新增配置项必须同步文档
* 配置初始化失败时应明确报错，而不是沉默降级

### 9.3 高风险配置

以下配置变更通常属于高风险：

* 认证与鉴权配置
* 数据库连接配置
* 消息队列 topic / consumer group
* 缓存 key 命名规则
* 定时任务开关
* 第三方服务 endpoint
* 超时、重试、熔断参数

## 10. 错误处理架构

### 10.1 总体原则

* 错误应带上下文
* 用户可见错误与内部错误分离
* 不允许通过吞错让流程“假装成功”
* 关键失败必须有日志与可观测性
* 跨模块错误传递应保留语义

### 10.2 错误分类建议

* 参数错误
* 业务规则错误
* 资源不存在
* 权限不足
* 依赖调用失败
* 存储失败
* 超时 / 取消
* 未知内部错误

## 12. 安全边界

本节用于标记哪些能力是高风险能力，哪些区域必须显式保护。

### 12.1 高风险能力示例

* 文件删除 / 覆盖
* 本地命令执行
* 配置动态变更
* 数据批量修改
* 权限变更
* 对外通知或回调
* 资金、库存、状态等关键资源写操作

### 12.2 安全原则

* 高风险操作必须显式确认或受控触发
* 权限边界必须清晰
* 外部输入必须校验
* 敏感数据必须脱敏
* 默认最小权限

### 12.3 AI Agent 限制

AI Agent 对以下区域应默认保守：

* 鉴权
* 计费 / 交易 / 状态机
* 批量任务
* 数据迁移
* 删除性操作
* 基础依赖初始化
* 部署与生产配置相关逻辑

## 14. AI Agent 可修改边界

为了让 AI Agent 参与优化时更安全，建议明确以下边界。

### 14.1 优先可改区域

* 文档
* 低风险工具模块
* 重复逻辑明显的辅助函数
* 错误处理不统一但影响面有限的区域
* 已有测试覆盖的局部模块
* 日志与可观测性补强
* 低风险配置整理

### 14.2 谨慎改动区域

* 核心业务服务
* 状态机
* 并发与异步逻辑
* 缓存策略
* 幂等逻辑
* 外部依赖适配层

### 14.3 默认不直接改动区域

* 鉴权与权限边界
* 核心数据模型
* 关键事务流程
* 核心对外接口协议
* 部署、发布、生产配置
* 删除性、高风险自动化能力

## 16. 架构收敛方向

本项目后续架构优化建议朝以下方向逐步收敛：

* 接口层更薄
* 业务层更清晰
* 外部依赖统一适配
* 错误处理统一
* 配置集中管理
* 关键路径可观测
* 高风险链路有测试保护
* 模块边界更稳定
* 新代码不继续加重旧结构混乱

### 收敛原则

* 不要求一次完成
* 每次只推进一个明确目标
* 先可验证，再推广
* 先低风险，后高风险
* 优先处理最阻碍开发效率和稳定性的区域
