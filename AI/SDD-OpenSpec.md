# OpenSpec 上手指南

只需遵循“安装 → 初始化 → 提案 → 审查 → 实施 → 归档”这六步，就能将它整合进你的开发流程中。

### 🚀 快速开始：安装与初始化

首先，确保你的开发环境满足以下要求：
*   Node.js >= **20.19.0**
*   一个受支持的 AI 编码助手（如 VS Code、Cursor 等，支持超过 25 种）

#### 1. 全局安装 CLI
在你的终端中执行以下命令来全局安装 OpenSpec 的命令行工具（CLI）：
```bash
npm install -g @fission-ai/openspec@latest
```

> 💡 **注意**：安装时如果遇到权限问题，你可能需要在命令前加上 `sudo`。此外，也可以使用 `pnpm`、`yarn` 等其他包管理器进行安装。

#### 2. 验证安装
安装完成后，执行以下命令以确保 OpenSpec 已正确安装并可用：
```bash
openspec --version
```
如果命令成功执行并返回版本号，则说明安装成功。

#### 3. 初始化你的项目
进入你的项目根目录，运行 `init` 命令来初始化 OpenSpec 环境：
```bash
cd your-project
openspec init
```
初始化过程会：
1.  询问你使用的 AI 工具（如 Cursor, Copilot 等），并自动为其生成对应的**斜杠命令**配置文件。
2.  在你的项目根目录下创建一个 `openspec/` 文件夹，包含 `project.md`（项目上下文）和 `AGENTS.md`（AI 助手说明书）等核心文件。

### 📂 理解核心结构：项目骨架

初始化完成后，你的项目根目录下会生成一个 `openspec/` 目录，这就是所有“规范”的存放地。理解它的结构至关重要：

| 文件/目录 | 作用 |
| :--- | :--- |
| `project.md` | **项目级规范**，用于定义项目的背景、技术栈和全局约定，是所有工作的基础。 |
| `specs/` | **“单一真相源”**，存放描述当前已实现功能的规范文件。 |
| `changes/` | **工作区**，存放所有正在进行中的变更提案，每个提案都是一个独立的子文件夹。 |
| `changes/[变更名]/` | 一个具体的变更提案，包含了它的需求、设计和任务。 |
| `changes/archive/` | **历史归档**，存放所有已完成并归档的变更记录。 |

### 📝 编写项目上下文

在开始第一个变更前，可以先编辑 `openspec/project.md` 文件，写下项目的核心目标和技术选型。这能帮助 AI 更好地理解你的项目全貌，避免在后续开发中偏离方向。例如：
```markdown
# Project Context
## Purpose
一个用于厨房的、在浏览器中运行的、简单且高可见性的计时器。

## Tech Stack
- HTML5
- JavaScript (ES6)
- CSS3
```

### 💡 核心四步工作流

掌握了基础，现在进入日常开发的“重头戏”。OpenSpec 的核心是“提案-审查-实施-归档”的工作流。

#### 1️⃣ 提案
当你需要新增或修改功能时，一切从一个提案开始。
*   **操作方式**：在 AI 助手中，使用 **`/opsx:new`** 命令创建一个新变更。
    ```bash
    /opsx:new add-dark-mode
    ```
*   **AI 执行的动作**：为你的新变更创建一个独立的文件夹（如 `changes/add-dark-mode/`）。在该文件夹下，你可以手动编写或继续使用 AI 命令生成以下文件：
    *   **`proposal.md`**：阐述变更的**动机、范围和影响**。它回答“**为什么要做**”和“**做什么**”的问题。
    *   **`design.md`**：描述实现变更的**技术方案和决策**，例如架构选择、数据流等。
    *   **`tasks.md`**：将方案拆解为**具体的、可验证的实施步骤清单**。
    *   **`specs/`**：存放**规范增量 (Delta Specs)**，以补丁的形式描述对现有规范的具体修改。

> 💡 **加速技巧**：你可以使用 **`/opsx:ff`** 命令让 AI 一次性快速生成 `proposal.md`、`design.md` 和 `tasks.md` 等所有规划文档。

#### 2️⃣ 审查与对齐
在 AI 生成所有规划文档后，**人工审查**是确保质量的关键一步。如果 AI 理解有误或你希望调整方案，可以直接在聊天中继续讨论，AI 会根据你的反馈修改相应文件。当你和 AI 都对提案达成一致后，就可以进入实施阶段了。

#### 3️⃣ 实施
一切就绪，现在让 AI 按照我们约定好的计划编写代码。
*   **操作方式**：在 AI 助手中，使用 **`/opsx:apply`** 命令启动实施。
*   **AI 执行的动作**：AI 会读取 `tasks.md` 中的任务清单，并**逐步完成每一项任务**，它会自动编写代码、运行测试，确保代码实现与规范一致。

#### 4️⃣ 归档
当 AI 完成所有任务，并且你对结果满意后，就可以将这次变更归档。
*   **操作方式**：在 AI 助手中，使用 **`/opsx:archive`** 命令进行归档。
*   **AI 执行的动作**：`changes/` 下的变更文件夹会被移动到 `changes/archive/` 目录中保存。同时，AI 会将该变更的规范增量合并更新到 `openspec/specs/` 目录下的主规范文件中，确保“单一真相源”始终反映项目的最新状态。

### 🛠️ 常用 CLI 命令速查

以下是你可能会经常用到的终端命令：

| 命令 | 用途 |
| :--- | :--- |
| `openspec list` | 查看当前所有“进行中”的变更及任务进度。 |
| `openspec show <id>` | 显示某个变更的详细信息（如 JSON 格式）。 |
| `openspec validate <id>` | 验证变更文档的格式是否正确（推荐加 `--strict` 参数）。 |
| `openspec archive <id> --yes`| 非交互式地将变更直接归档。 |
| `openspec update` | 升级 OpenSpec 后，更新 AI 配置文件到最新模板。 |

### 🤝 如何在现有项目中使用？

OpenSpec 为“棕地项目”（即存量项目）设计了特殊的集成路径，能够无缝融入现有开发流程。

*   **对接企业Jira**：使用 `/opsx:propose DSH-618 增加登录验证码` 命令，可以将 Jira 工单作为上下文，创建一个变更提案。
*   **逆向生成SDD文档**：对于已完成的代码，可以使用 `/opsx:sdd-docs DSH-618 修复登录验证码异常` 命令，基于当前的代码差异自动生成规范文档，用于代码审查和归档。

### ✨ 进阶技巧与最佳实践

*   **📝 善用 `/opsx:ff` (Fast-Forward)**：这是提升效率的关键。在进行常规功能开发时，先用 `/opsx:new` 创建变更，紧接着使用 `/opsx:ff` 让 AI 一次性生成所有规划文档，能节省大量时间。

*   **🔧 规范是活的，随时可改**：在开发过程中，如果你对 `proposal.md` 或 `design.md` 的想法有变，**随时**告诉 AI，它可以立刻更新，没有僵化的阶段限制。

*   **✔️ 将 `/opsx:verify` 集成到 CI 中**：OpenSpec 的验证能力可以整合到你的持续集成流水线中，确保所有代码合并请求都通过了规范验证，从流程上保障开发质量。

*   **🔄 善用 `openspec update` 维护环境**：当你的 OpenSpec CLI 工具升级后，及时运行 `openspec update` 命令，它会将最新的 AI 指令和模板更新到你现有的配置文件（如 `.cursor/commands/`）中。
