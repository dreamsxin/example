### 二、技术难点速查

1. **幻觉控制** — RAG 检索精度不足 / 知识库质量差 → 方案：HyDE、重排序、知识图谱增强
2. **长上下文处理** — 超长文档如何保持召回精度 → 方案：Small-to-Big 检索、层级索引
3. **多轮对话一致性** — 上下文窗口管理、记忆压缩 → 方案：摘要记忆 + 向量记忆混合
4. **推理延迟优化** — 首 Token 延迟 vs 吞吐量权衡 → 方案：Continuous Batching、Speculative Decoding
5. **Prompt 稳定性** — 同 Prompt 不同模型表现差异大 → 方案：DSPy 自动优化 + 回归测试
6. **安全对齐** — 越狱攻击、敏感内容过滤 → 方案：多层护栏（输入+输出）

### 三、常见面试问答速记

**Q1: Transformer 的注意力机制为什么要除以 √d_k？**
> 点积方差随维度增大 → Softmax 梯度消失 → 除以 √d_k 将方差稳定在 1，保持梯度健康。

**Q2: LoRA 的原理和优势？**
> 冻结原权重，注入低秩矩阵 A·B 作为增量更新。优势：显存省 90%+，多任务可热切换，无推理延迟。

**Q3: RAG 中召回率低怎么优化？**
> ① 混合检索（稀疏 BM25 + 稠密向量）② Query 重写/扩展 ③ HyDE 假设文档嵌入 ④ 重排序模型 (Cross-Encoder)

### 第 1 题

> "请描述一个你从 0 到 1 落地的大模型项目。当时为什么选择大模型方案而不是传统 NLP？你如何评估项目是否成功？"

**S — Situation**

公司内部客服系统日均处理 3000+ 工单，人工客服压力大，且历史知识库（2 万+ 篇文档）利用率极低。传统 NLP 方案（规则匹配 + 意图分类）覆盖率仅 40%，用户问题稍微变种就匹配不上。

**T — Task**

从 0 搭建一个基于 LLM 的智能客服系统：自动理解用户问题、从知识库检索相关文档、生成准确回答。要求上线后自动解决率 ≥ 65%，回答准确率 ≥ 90%。

**A — Action**

| 阶段 | 具体动作 |
|------|----------|
| **选型** | 对比了 GPT-4、Qwen-72B、DeepSeek-V2，最终选 Qwen-72B（私有化部署 + 中文能力强 + 成本可控） |
| **RAG 链路** | 文档 → 按标题/段落切分（512 token chunk + 128 overlap）→ BGE-M3 Embedding → Milvus 向量库 |
| **检索优化** | BM25 + 向量混合检索 → BGE-Reranker 重排序 → Top-5 召回 |
| **Prompt 工程** | 严格 System Prompt（角色 + 边界 + 拒答规则）+ 动态拼接历史对话 |
| **上线策略** | 先灰度 10% 流量 → A/B 对比 → 人工抽检 500 条 → 逐步放量至 100% |

**R — Result**

- 自动解决率：**68%**（超目标 3pp）
- 回答准确率：**92.3%**（人工抽检）
- 人工客服日均工单：3000 → **960**（降 68%）
- P99 首 Token 延迟：**1.2s**

**为什么选 LLM 而不是传统 NLP？**
- 传统方案每个意图都要写规则/训分类器，知识库更新就要重训，维护成本不可接受
- LLM 零样本泛化能力强，用户换种说法也能理解
- RAG + LLM 天然解决了"知识随时更新"的需求

### 第 2 题

> "如果你的 RAG 系统在生产环境中出现大量'幻觉'回答，用户投诉严重，你会如何系统性排查和解决？"

**排查路线图（从外到内，逐层定位）：**

```
用户举报 → 抽样分析 → 分层定位 → 逐个修复 → 回归验证
```

**第一层：输出层定位**

| 检查项 | 判断标准 | 典型问题 |
|--------|----------|----------|

| 幻觉类型 | 编造数据？张冠李戴？过度推理？ | 不同类型根因不同 |
| 是否拒答 | 应该拒答但没拒答？ | System Prompt 边界不够严 |

```python
    has_answer = any(contains_answer(doc, case.expected) for doc in docs)

    # 如果 80% 幻觉案例"文档里有答案但 LLM 没用"→ 生成问题
```

**第三层：根因修复（按命中率排序）**

| 根因 | 占比（经验） | 修复方案 |
|------|-------------|----------|
| 🔴 检索没召回正确文档 | ~45% | ① 加 BM25 混合检索 ② Query 改写（LLM 扩展同义词） ③ 调高 chunk overlap |
| 🔴 召回了但排序靠后被截断 | ~20% | ① 加入 BGE-Reranker ② Top-K 从 5 扩到 10 |
| 🟡 文档切分破坏了语义 | ~15% | ① 按段落/标题切分替代固定长度 ② 尝试 Small-to-Big（检索小 chunk，喂大 context） |
| 🟡 System Prompt 太弱 | ~10% | ① 加入"如果上下文不包含答案，请明确说不知道" ② 加入 Few-shot 示例 |
| 🟢 LLM 自身问题（温度过高等） | ~10% | ① temperature 降至 0.1 ② 换更好的模型 |

**上线后闭环：**
- 建立"幻觉标注 → 每周分析 → 持续优化检索/切分策略"的飞轮
- 增加在线评估（LLM-as-Judge 自动打分 + 人工抽检）

### 第 3 题

> "请比较 LoRA、QLoRA 和 Full Fine-tuning 的适用场景。如果给你 100 万条高质量领域数据 + 8×A100，你会选哪种策略？为什么？"

**三者对比：**

| 维度 | Full Fine-tuning | LoRA | QLoRA |
|------|:--:|:--:|:--:|
| **显存需求** | 极高（72B 需 ~576GB） | 低（72B ~80GB） | 极低（72B ~40GB） |
| **训练速度** | 🔴 慢 | 🟢 快（参数量少） | 🟡 中（量化开销） |
| **模型性能上限** | 🟢 最高 | 🟡 略低于全参（~95-98%） | 🟡 同 LoRA |
| **灾难性遗忘** | 🔴 高 | 🟢 低（原权重冻结） | 🟢 低 |
| **多任务切换** | 🔴 需保存完整模型 | 🟢 只换 LoRA 权重（~百 MB） | 🟢 同 LoRA |
| **适用数据量** | 百万+ 高质量 | 千~十万 | 千~十万 |
| **硬件门槛** | 8×A100 起步 | 单卡 A100/4090 | 单卡 3090/4090 |

**我的选择：Full Fine-tuning（全参微调）**

理由：

1. **100 万条高质量数据不是瓶颈** — LoRA 的优势是少数据场景，但 100 万条高质量数据已经满足全参微调的数据需求，用 LoRA 反而浪费了数据优势

2. **8×A100 硬件充足** — 以 Qwen-72B 为例，全参微调约需 ~600GB 显存，8×A100(80G) = 640GB，刚好够用（配合 DeepSpeed ZeRO-3 + FlashAttention-2 + Gradient Checkpointing）

3. **领域知识需要深度注入** — 如果是医疗/法律/金融等强专业领域，LoRA 的低秩约束可能不足以捕获全部领域知识，全参微调的上限更高

4. **实际训练配置：**
   ```
   框架：DeepSpeed ZeRO-3 + FlashAttention-2
   精度：BF16 混合精度
   学习率：2e-5，cosine schedule，warmup 10%
   Batch size：global batch = 128（每卡 16 × 8 卡 × grad_acc 1）
   Epoch：2-3 轮
   ```

**什么情况下会选 LoRA/QLoRA？**
- 数据 < 5 万条 → LoRA（全参会严重过拟合）
- 只有 1-2 张卡 → QLoRA
- 需要快速实验多种超参 → LoRA（训练快，换权重成本低）
- 需要同时服务多个垂直场景 → LoRA（一个基座 + 多个 LoRA 模块热切换）

### 💡 面试回答技巧提醒

| 要点 | 说明 |
|------|------|
| **先说结论** | 面试官注意力有限，先给答案再展开 |
