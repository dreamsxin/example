# Project Name

> 一个正在持续演进的 Golang 项目。  
> 本项目已接入 AI Agent 辅助开发，目标是在**不破坏现有业务稳定性**的前提下，逐步完成结构治理、质量提升、性能优化与文档完善。

---

## 1. 项目背景

本项目是一个已有的 Golang 系统，目前重点不是从零开发，而是对既有工程进行持续优化，包括但不限于：

- 代码结构整理
- 历史模块重构
- 测试补齐
- 性能优化
- 错误处理统一
- 配置治理
- 日志与可观测性增强
- 文档补全
- AI Agent 辅助改造

---

## 2. 当前目标

当前阶段的核心目标：

- 保持业务可用性
- 降低维护成本
- 提高可测试性
- 降低模块耦合
- 让 AI Agent 能够安全、稳定地参与开发

非目标：

- 一次性推翻重写
- 未经验证的大规模替换
- 为了“看起来更优雅”而破坏线上稳定性

---

## 3. AI Agent 参与原则

本项目允许 AI Agent 参与分析、重构、补测试、补文档和局部优化，但必须遵守以下原则：

### 3.1 优先级原则

AI 的工作优先级如下：

1. **不破坏现有行为**
2. **先理解再修改**
3. **先小步修改再扩大范围**
4. **先补测试再重构**
5. **优先局部治理，避免全局重写**

### 3.2 禁止行为

AI Agent 禁止：

- 未经说明直接大规模重构
- 擅自修改公共接口
- 擅自删除历史逻辑
- 假设不存在的上下文
- 引入未经批准的新依赖
- 伪造测试结果
- 输出不能运行的示意代码
- 用“TODO”代替实现
- 为了通过编译而绕过错误处理

### 3.3 每次输出必须包含

AI 每次改动必须明确给出：

1. 修改目标
2. 修改原因
3. 涉及文件
4. 风险影响
5. 回滚方式（如适用）
6. 完整文件或可直接应用的 patch

---

## 4. 适用的优化类型

AI Agent 可以做的事包括：

- 分析模块职责
- 提出重构拆分方案
- 提取重复逻辑
- 统一错误处理
- 增加单元测试
- 增加集成测试
- 梳理配置项
- 补日志埋点
- 提升可观测性
- 清理死代码
- 优化 SQL / I/O / 并发逻辑
- 补充 README / API 文档 / 维护文档

AI Agent 不应直接做的事包括：

- 不经验证改写核心交易链路
- 不经审批替换核心中间件
- 修改生产基础设施配置
- 自动迁移数据库结构
- 直接变更安全策略

## 5. 推荐目录约定

如果当前项目历史包袱较重，可逐步向以下结构收敛：

```text
.
├── cmd/                    # 程序入口
├── internal/               # 内部业务代码
│   ├── app/                # 应用编排层
│   ├── service/            # 业务服务层
│   ├── domain/             # 领域模型与规则
│   ├── repository/         # 数据访问层
│   ├── transport/          # HTTP/gRPC/消息队列等接口层
│   ├── config/             # 配置加载
│   ├── middleware/         # 中间件
│   ├── errors/             # 统一错误定义
│   ├── logging/            # 日志封装
│   └── utils/              # 内部工具函数
├── pkg/                    # 可对外复用的稳定库（如确有必要）
├── api/                    # 接口定义、OpenAPI、proto 等
├── configs/                # 配置模板
├── scripts/                # 脚本
├── test/                   # 集成测试、fixture、测试资源
├── docs/                   # 文档
├── deployments/            # 部署配置
├── go.mod
├── go.sum
└── README.md
````

说明：

* **新代码优先进入 `internal/`**
* `pkg/` 只放明确可复用、稳定的公共代码
* 不允许把所有逻辑继续堆在 `cmd/` 或 `handler/` 中

---

## 6. 分层约束

### 6.1 transport 层

职责：

* 接收请求
* 参数校验
* 调用 service
* 返回响应

禁止：

* 在 handler/controller 中写复杂业务逻辑
* 在 transport 层直接拼接 SQL
* 在 transport 层处理跨模块事务

### 6.2 service 层

职责：

* 编排业务流程
* 调用 repository / domain
* 组织事务边界
* 处理业务规则

禁止：

* 把所有业务都写成超长函数
* 混杂基础设施细节
* 把日志、配置、缓存、DB 代码耦合成一团

### 6.3 repository 层

职责：

* 数据访问
* 查询封装
* 存储适配

禁止：

* 在 repository 中写业务规则
* 返回模糊不清的错误
* 在多个地方复制粘贴相似 SQL

### 6.4 domain 层

职责：

* 核心领域对象
* 领域规则
* 与存储和协议无关的业务逻辑

禁止：

* 依赖 HTTP / gRPC / DB 细节
* 耦合外部库实现

---

## 7. 重构策略

对已有项目进行 AI 优化时，必须遵循以下顺序：

### 第一步：理解现状

* 梳理模块边界
* 找出核心调用链
* 找出高风险路径
* 标记脏代码与重复逻辑

### 第二步：建立保护

* 先补测试
* 先增加日志
* 先补充文档
* 先确认输入输出行为

### 第三步：局部治理

* 先小函数拆分
* 先提取重复逻辑
* 先统一错误处理
* 先把魔法字符串/硬编码集中治理

### 第四步：逐步收敛架构

* 调整目录
* 收敛依赖方向
* 抽出 service/repository/domain
* 提高模块独立性

### 第五步：性能与可观测性优化

* 指标监控
* profile 分析
* SQL 优化
* 并发控制
* 缓存策略

## 8. 代码规范

### 8.1 Go 基本规范

* 必须通过 `gofmt`
* 必须通过 `go vet`
* 推荐通过 `golangci-lint`
* 函数职责单一
* 避免超长函数和超大文件
* 避免包循环依赖

### 8.2 错误处理规范

* 不忽略错误
* 不允许生产路径滥用 `panic`
* 错误信息应包含上下文
* 对外暴露错误与内部错误分离
* 必要时使用 `errors.Is` / `errors.As`

### 8.3 日志规范

* 关键路径必须有日志
* 日志必须可搜索
* 禁止打印敏感信息
* 错误日志必须带上下文
* 请求链路建议带 trace / request id

### 8.4 配置规范

* 禁止硬编码环境差异
* 所有配置集中管理
* 提供默认值与校验
* 敏感配置走环境变量或密钥系统

---

## 9. 测试要求

### 最低要求

* 新增逻辑必须有测试
* 修 bug 必须补回归测试
* 重构前应先补保护性测试

### 测试类型

* 单元测试：业务规则、工具函数、service 方法
* 集成测试：DB、缓存、外部依赖协作流程
* 回归测试：历史 bug 场景
* 基准测试：热点逻辑性能

### 禁止

* 只补“形式测试”
* mock 一切导致测试失真
* 不验证关键业务结果

---

## 10. 性能优化约束

AI Agent 可以进行性能优化，但必须：

1. 先给出瓶颈判断依据
2. 说明优化假设
3. 明确影响范围
4. 有 benchmark / profile / 指标支撑时优先
5. 不以可读性彻底崩坏为代价

常见优化方向：

* 减少重复分配
* 优化热路径
* 控制 goroutine 数量
* 减少锁竞争
* 批量化 I/O
* 优化 SQL 与索引使用
* 减少序列化/反序列化损耗

---

## 11. 兼容性原则

已有项目经常涉及兼容性问题，AI Agent 修改时必须评估：

* 是否影响对外 API
* 是否影响历史数据
* 是否影响配置项
* 是否影响数据库 schema
* 是否影响日志采集
* 是否影响监控指标
* 是否影响上下游服务

若有兼容性风险，必须显式写出。

# AI Agent Rules

本文件定义 AI Agent 在本项目中的行为边界。  
优先级高于普通开发建议。

---

## 1. 总原则

AI Agent 的职责是帮助项目做**渐进式优化**，不是替代架构师进行无边界重写。

目标：

- 保持现有行为
- 降低维护成本
- 提升可测试性
- 提升可观测性
- 提高代码清晰度
- 提供安全、可验证的修改

---

## 2. 先分析，后修改

在修改任何已有代码前，AI 必须先做以下动作：

1. 理解该模块输入输出
2. 确认调用方与被调用方
3. 判断是否属于核心链路
4. 判断是否已有测试覆盖
5. 标出潜在风险点

如果以上信息不足，优先给出“保守修改方案”。

---

## 3. 渐进式修改原则

允许的改法：

- 提取小函数
- 提取重复逻辑
- 统一错误处理
- 补充参数校验
- 增加日志
- 增加测试
- 小范围模块拆分
- 清理明显死代码

谨慎进行的改法：

- 调整包结构
- 变更接口定义
- 替换并发模型
- 替换中间件
- 重写 SQL 层
- 改变事务边界

禁止的改法：

- 一次性大重写
- 未说明原因删除历史逻辑
- 未验证就改核心链路
- 未审阅就新增大量依赖

---

## 4. 输出格式要求

每次输出修改方案时，必须包含：

### 4.1 修改说明
- 改了什么
- 为什么改
- 涉及哪些文件

### 4.2 风险说明
- 是否有行为变化风险
- 是否影响兼容性
- 是否影响性能
- 是否影响测试

### 4.3 交付格式
优先顺序：

1. 完整文件
2. 可直接应用的 patch
3. 分步骤提交建议

默认不要只输出片段，除非明确要求。

---

## 5. Go 项目约束

### 5.1 错误处理
- 不忽略 error
- 不在业务路径使用 panic
- 不允许用空返回值掩盖失败
- 错误必须带上下文

### 5.2 包结构
- 新代码优先进入 `internal/`
- 避免循环依赖
- 避免把工具函数乱放到公共包
- 避免巨型 util 包

### 5.3 接口设计
- 优先最小接口
- 不为了 mock 而抽象一切
- 不无意义拆接口
- 不新增多层无价值包装

### 5.4 并发
- goroutine 必须可控
- 必须考虑退出、超时、取消
- 共享状态必须考虑同步
- 避免 goroutine 泄漏

### 5.5 上下文传递
- I/O 边界优先传 `context.Context`
- 不把 context 存 struct
- 不滥用 `context.Background()`

---

## 6. 测试规则

修改代码时必须判断是否要补测试：

### 必须补测试的情况
- 修 bug
- 重构核心逻辑
- 增加分支逻辑
- 调整错误处理
- 调整并发逻辑

### 测试要求
- 验证行为，不只是跑通
- 尽量覆盖边界条件
- 避免与实现细节强耦合
- 回归测试要能复现历史问题

---

## 7. 文档规则

以下修改应同步文档：

- 新配置项
- 新接口
- 目录结构调整
- 新的运行方式
- 新的限制条件
- 高风险模块变更

---

## 8. 性能优化规则

做性能优化前，必须至少满足一个条件：

- 有 profile 证据
- 有 benchmark 数据
- 有明确热点路径
- 有线上指标反馈

禁止：

- 无依据进行“想当然优化”
- 为微小收益引入复杂度爆炸
- 在不解释的情况下改动关键并发逻辑

---

## 9. 安全规则

禁止：

- 输出真实密钥
- 建议关闭鉴权以图方便
- 建议关闭错误处理
- 建议删除审计日志
- 默认扩大权限范围

必须：

- 路径访问要校验
- 配置要分环境
- 敏感日志要脱敏
- 对高风险操作给出保护方案

---

## 10. 与人类协作方式

AI Agent 不是最终裁决者。

对于以下情况，AI 应明确标记风险，而不是擅自决定：

- 公共 API 改动
- 数据库 schema 变更
- 核心交易链路重构
- 依赖替换
- 安全策略调整
- 大规模目录迁移

---

## 11. 默认工作顺序

遇到一个已有模块时，AI 默认按以下顺序工作：

1. 理解现状
2. 描述问题
3. 标记风险
4. 给出最小修改方案
5. 补测试
6. 再考虑进一步重构
7. 最后考虑性能优化

---

## 12. 一句话要求

> 小步修改、保持兼容、先测后改、先稳后优。
````

# 三、适合已有 Go 项目的目录补强模板

如果你的老项目比较乱，可以加一个“目标结构说明”，给 AI 明确收敛方向：

```text
internal/
  app/            # 应用装配与启动编排
  service/        # 业务服务
  domain/         # 领域对象与规则
  repository/     # DB/缓存/外部存储
  transport/      # http/grpc/message handler
  config/         # 配置加载与校验
  logging/        # 日志封装
  errors/         # 统一错误
  middleware/     # 中间件
  util/           # 少量内部工具
```

给 AI 的约束可以写成一句：

> 新逻辑优先往目标结构迁移，不要求一次完成，但不能继续加重旧结构混乱。

# 四、最实用的 AI 任务模板

下面这些提示词非常适合“优化已有 Go 项目”。

## 1. 梳理模块

```text
请分析这个已有 Golang 模块的职责、输入输出、依赖关系、重复逻辑和潜在风险。
不要重写，只做结构理解和优化建议。
输出格式：
1. 当前职责
2. 存在问题
3. 风险点
4. 最小优化方案
5. 后续可选优化
```

## 2. 安全重构

```text
请对这个已有 Golang 文件做渐进式重构。
要求：
1. 不改变外部行为
2. 先提取重复逻辑
3. 拆分超长函数
4. 保持接口兼容
5. 补充必要错误处理
6. 输出完整文件
```

## 3. 补测试

```text
请为这个已有 Golang 模块补充测试。
要求：
1. 优先覆盖当前行为
2. 覆盖边界条件和错误分支
3. 如存在历史 bug 风险，增加回归测试
4. 不要为了测试强行重构生产代码，除非确有必要
```

## 4. 性能优化

```text
请评估这段 Golang 代码的性能风险。
要求：
1. 先说明可能的瓶颈
2. 再给出优化建议
3. 如果值得修改，再输出完整优化代码
4. 不要做无证据的大改
```

---

# 五、你这个模板的核心价值

这个模板最适合解决 4 个常见问题：

1. **AI 一上来就想重写**

   * 通过“渐进式优化”规则压住

2. **AI 乱改接口**

   * 通过“兼容性评估”限制

3. **AI 只会说思路，不给可用代码**

   * 通过“必须输出完整文件 / patch”约束

4. **AI 在老项目里越改越乱**

   * 通过“目标结构收敛 + 新代码迁移规则”控制
